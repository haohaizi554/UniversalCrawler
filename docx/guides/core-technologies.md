# 核心技术参考

> 面向后续维护者的技术地图：说明本项目用了什么、为什么这样分层、扩展时应该守住哪些边界。

## 一句话架构

Universal Crawler Pro 是一个以 PyQt6 桌面端为主入口、FastAPI WebUI 和 CLI/SDK 为辅助入口的多平台媒体采集系统。核心链路是：

```text
入口层 -> 应用控制层 -> 平台插件 -> 资源解析 -> 任务构建 -> 下载调度 -> 本地媒体管理
```

GUI、WebUI、CLI 不应该各自实现一套业务逻辑。它们只负责收集输入、展示状态和触发命令，真正的配置、平台解析、任务构建、下载和日志能力应尽量复用后端服务。

## 技术栈总览

| 方向 | 核心技术 | 在项目中的职责 |
| --- | --- | --- |
| 桌面端 | PyQt6 | 主窗口、设置中心、任务队列、媒体预览、日志中心、弹窗和系统集成 |
| Web 服务 | FastAPI、Uvicorn | WebUI/API、远程控制、容器模式、无 GUI 服务入口 |
| 浏览器自动化 | Playwright Chromium | 动态页面访问、登录态页面解析、复杂平台采集辅助 |
| 网络请求 | requests、httpx、curl-cffi | 常规 HTTP 请求、异步请求、浏览器指纹兼容请求 |
| 下载执行 | DownloadManager、下载器适配层、外部工具 | 队列调度、进度同步、HLS/直链/外部工具下载 |
| 外部工具 | ffmpeg、N_m3u8DL-RE、yt-dlp | 合并、转封装、HLS 下载、特殊站点下载兜底 |
| 配置管理 | ConfigManager、FrontendStateService | 运行时设置、热加载快照、GUI/WebUI 配置契约 |
| 平台扩展 | 插件注册表、BasePlugin、Spider | 平台识别、输入解析、资源抓取、任务生成 |
| 日志观测 | loguru、日志中心、调试追踪 | 用户可见日志、开发排障、运行链路审计 |
| 打包部署 | PyInstaller、Inno Setup、Docker | 桌面便携版、安装包、Web 服务容器化 |
| 测试质量 | pytest、test_launcher、Playwright 测试环境 | 单元测试、集成测试、回归测试、分组运行 |

## 入口层

入口层位于 `main.py` 和 `entry/`：

- `main.py` 是自适应启动入口，按参数、环境和运行环境分发到 GUI、Web、CLI 或测试入口。
- `entry/gui_entry.py` 启动桌面 GUI。
- `entry/web_entry.py` 启动 FastAPI Web 服务，可带 Qt 托盘，也可用 `--no-qt --no-browser` 进入无界面模式。
- `entry/cli_entry.py` 提供命令行入口。
- `entry/dispatcher.py` 负责统一的模式选择和参数分发。

维护原则：

- 不要把业务逻辑写死在入口脚本里。
- 新增入口参数时，要确认 GUI、Web、CLI 是否需要共享同一份配置来源。
- 入口层应只做环境准备、模式选择、错误兜底和用户提示。

## GUI 技术路线

GUI 基于 PyQt6，承担项目最完整的用户体验：

- 左侧任务导航、平台选择、任务列表、日志中心、配置中心和工具箱。
- 下载队列、失败列表、完成列表、媒体预览、文件信息展示。
- 设置中心负责主题、语言、代理、下载策略、日志策略等用户可见选项。
- 弹窗、默认打开方式、系统托盘、文件选择等 Windows 桌面交互也在这一层处理。

维护原则：

- GUI 控件状态应尽量来自后端配置快照，而不是散落在控件自身。
- 主题、语言、字体、代理等设置要支持运行时更新，避免要求用户重启才能看到 UI 变化。
- 涉及 PyQt 信号、线程、定时器和下载回调时，必须明确对象生命周期，避免异常退出和 UI 卡顿。
- 控件样式要通过统一的主题变量和组件封装收敛，避免每个页面写一份独立样式。

## WebUI 与 API

Web 侧基于 FastAPI 和 Uvicorn，既服务 WebUI，也提供远程控制能力：

- Web 服务可单独运行，适合服务器、容器和自动化场景。
- WebUI 应与 GUI 对齐配置语义，尤其是平台、主题、语言、代理和下载策略。
- API 层负责把前端动作转成后端命令，不应该复制平台解析或下载逻辑。
- 容器模式默认禁用 Qt 和自动打开浏览器，避免桌面依赖污染服务端环境。

维护原则：

- WebUI 和 GUI 使用同一份后端设置契约。
- 新增可见文案时，应同步考虑国际化。
- 新增 API 时，应明确返回结构、错误码和前端空状态。
- 静态资源缓存、前端状态恢复和后端快照要一起验证，避免刷新后状态漂移。

## 平台插件与采集链路

平台扩展围绕插件注册表、平台 Spider 和资源模型展开：

```text
输入识别 -> 平台插件匹配 -> Spider 抓取 -> 资源解析 -> VideoItem/MediaItem -> 下载任务
```

内置平台包括：

- `douyin`
- `xiaohongshu`
- `kuaishou`
- `bilibili`
- `missav`

维护原则：

- 新平台优先新增插件和 Spider，不要把平台判断写进 GUI 或下载器。
- Spider 负责平台访问和解析，下载器负责拿到明确资源后的下载执行。
- 平台风控、Cookie、代理和浏览器上下文属于易变点，需要保留清晰日志。
- 不同平台返回数据不一致时，应先归一化到项目内部数据模型，再交给任务队列。

## 下载调度与外部工具

下载链路的核心目标是稳定、可观测、可恢复：

- DownloadManager 管理下载队列、并发、状态、进度和失败结果。
- 下载器适配层处理直链、HLS、外部工具和特殊站点策略。
- `ffmpeg` 用于合并、转封装和媒体处理。
- `N_m3u8DL-RE` 可用于 HLS 下载和合并。
- `yt-dlp` 可作为部分平台或格式的下载兜底。

维护原则：

- “只下载视频”“跳过封面和图片”等策略必须在任务构建和下载执行两端都生效。
- 外部工具不可用时要给出可读错误，不要让 GUI 静默失败。
- 下载失败应进入失败列表或可诊断状态，不能直接吞掉异常。
- 长耗时任务不要阻塞 UI 线程。

## 配置、主题与国际化

配置中心是 GUI、WebUI 和后端之间最容易发生漂移的区域。核心思想是：用户看到的设置项，必须有稳定的后端契约。

关键职责：

- 保存用户偏好和运行策略。
- 生成前端可消费的设置快照。
- 支持主题、语言、字体、代理、平台默认值等设置热加载。
- 保证 GUI 和 WebUI 对同一设置的命名、默认值和生效范围一致。

维护原则：

- 不要在 GUI 控件里私自定义一套默认值。
- 不要在 WebUI 单独维护一份与后端不一致的设置枚举。
- 主题色、深浅主题、字体大小和语言切换要避免整页白屏式重绘。
- 新增设置项时，应同步更新配置文档、前端映射、测试和默认值迁移逻辑。

## 日志与排障

项目需要同时服务普通用户和开发维护者，因此日志分为两类：

- 用户可见日志：任务进度、失败原因、配置提示、下载状态。
- 开发排障日志：平台解析、网络请求、外部工具调用、异常栈、状态迁移。

维护原则：

- 关键失败必须有上下文，比如平台、输入、代理、工具路径、HTTP 状态或异常类型。
- 不要把 Cookie、账号、完整个人路径等敏感信息写入公开日志。
- 具有教学意义的异常和修复，应记录到 `docx/` 下的 Markdown 文档。
- 日志中心 UI 要跟随主题和语言，不应出现浅色控件混入深色主题的情况。

## 打包与部署

项目同时支持桌面分发和 Web 服务部署：

- PyInstaller 用于生成便携版桌面程序。
- Inno Setup 用于生成 Windows 安装包。
- Docker 用于运行无 GUI 的 Web 服务。
- `docker-compose.yml` 默认挂载 `user_data/`、`downloads/` 和 `tools/`。

维护原则：

- 新增资源文件、图标、静态前端产物或外部工具目录时，要同步打包脚本。
- GUI 功能不应强行污染 Web 容器模式。
- 文件关联、默认打开方式等 Windows 功能要明确是立即生效、重启生效，还是下次打开生效。
- 发布前要验证开发模式、便携模式、安装包模式和 Docker 模式的入口差异。

## 测试与质量门禁

测试体系以 pytest 为基础，并通过测试启动器做分组管理：

- 单元测试覆盖模型、配置、工具函数和独立服务。
- 集成测试覆盖入口、Web API、CLI/SDK、下载链路和配置契约。
- 浏览器相关测试依赖 Playwright Chromium。
- GUI 相关测试要注意 QApplication 生命周期和控件线程边界。

维护原则：

- 修复用户可见 bug 时，应优先补回归测试。
- 新增设置项时，应测试默认值、保存、热加载、GUI/WebUI 同步。
- 新增平台插件时，应至少覆盖输入识别、资源模型和失败兜底。
- 文档改动不需要强行跑全量测试，但影响入口、配置、下载、平台解析时必须跑相关分组。

## 新人扩展清单

新增平台时：

1. 在插件层声明平台元数据、图标、默认配置和识别规则。
2. 实现 Spider，处理关键词、链接、主页等输入。
3. 把平台返回值归一化为内部资源模型。
4. 接入任务构建和下载链路。
5. 补充测试、README 平台表和必要排障文档。

新增设置项时：

1. 定义后端默认值和存储结构。
2. 更新设置快照和前端契约。
3. 同步 GUI 控件、WebUI 控件和国际化文案。
4. 验证热加载、重启后恢复和异常值回退。
5. 更新 `guides/config.md` 或相关专项文档。

新增下载策略时：

1. 明确策略属于任务构建阶段还是下载执行阶段。
2. 更新 DownloadManager 或下载器适配层。
3. 处理外部工具不存在、下载中断、格式不支持等失败路径。
4. 在日志中心展示可理解的状态和原因。
5. 补充回归测试和排障说明。

## 最重要的工程约束

- GUI、WebUI、CLI 共享核心业务，不做三套平行实现。
- 平台解析和下载执行分层，不让 Spider 直接承担下载器职责。
- 设置项以统一后端契约为准，不让控件默认值成为隐形事实。
- 长耗时任务必须离开 UI 线程。
- 外部工具和平台策略都不稳定，错误信息要可诊断。
- 文档要跟着重要变更一起更新，否则后续维护成本会快速上升。
