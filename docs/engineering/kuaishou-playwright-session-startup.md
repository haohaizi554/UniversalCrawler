# 快手 Playwright 会话启动与闪窗排障实践

> 本文记录快手任务启动时短暂出现额外 Chrome 窗口的根因、诊断方法和当前推荐实现。相关代码位于 `app/spiders/kuaishou/spider.py`。

更新日期：2026-07-16

## 问题结论

“任务启动时先闪过一个小窗，再出现正式浏览器”并不是快手输入类型、Node.js 控制台或用户误操作导致的。根因是有头模式创建 `BrowserContext` 时，Playwright 为恢复 `storage_state.origins` 中的 localStorage，会执行一段内部导航流程：

1. 创建临时页面或窗口。
2. 导航到保存态中的 origin。
3. 写入 localStorage 等站点存储。
4. 关闭临时页面。
5. 应用代码随后再创建真正的任务页面。

因此，只要本地存在包含 `origins` 的快手登录态，任何使用有头模式的快手任务都可能出现闪窗，与关键词、主页链接、用户链接或分享链接无关。

## 如何确认根因

只观察 UI 很容易把临时窗口误判为登录窗口、反检测脚本或 Chromium 启动器。排查时应同时记录窗口句柄和进程：

- 用 `EnumWindows` 枚举可见顶层窗口。
- 用 `GetWindowThreadProcessId` 取得窗口所属 PID。
- 只记录本次启动后新增的 `chrome.exe` / `chromium.exe` 窗口。
- 以 HWND 为身份，而不是只看窗口标题；标题会从“无标题”变为 `about:blank`，再变为站点标题。

旧实现的典型序列是：同一 Chrome PID 下出现 HWND A，A 很快消失，随后出现 HWND B。修复后的有头任务应始终只有一个 HWND，其标题可以变化，但句柄不应发生“关闭后重建”。

还应先排除 Playwright 驱动进程本身。Windows 版 Playwright 的 `PipeTransport` 已使用隐藏窗口标志启动 Node.js；如果监控到的是 Chrome 顶层 HWND，就不应继续从控制台窗口方向猜测。

## 当前恢复策略

有头会话与无头会话需要采用不同策略。

### 有头会话

有头会话创建 context 时只把 Cookie 交给 Playwright，并将 `origins` 置空。随后在创建第一个真实页面之前，通过 context 级 `add_init_script()` 按 `location.origin` 恢复 localStorage。

这样既保留登录 Cookie 和站点本地状态，又不会触发 Playwright 的内部 origin 恢复页。初始化脚本必须满足以下约束：

- 在 `context.new_page()` 之前注册。
- 只恢复快手和其已授权关联域名，不能把任意 origin 写入浏览器。
- 值按字符串原样恢复，不解析或重新序列化业务内容。
- 无有效 origin 存储时不注入空脚本。

### 无头会话

无头会话没有可见闪窗问题，应继续把完整 `storage_state` 交给 Playwright。原因是未来或当前登录态可能包含 IndexedDB；仅靠 localStorage 初始化脚本无法等价恢复 IndexedDB。

不要为了统一代码路径，让无头会话也退化为“Cookie + localStorage”。这会在表面消除一个不存在的 UI 问题，却可能悄悄破坏静默采集的认证状态。

## 分享链接必须独立分流

分享或单条详情链接与主页、用户页、搜索页不是同一种任务。当前推荐流程是：

```text
规范化输入与短链重定向
  -> 判断是否为分享/详情链接
    -> 纯 HTTP 快速解析
      -> 成功：提交一个下载任务并结束
      -> 失败：进入固定无头的详情页捕获会话
    -> 普通任务：按设置进入登录检查、目标页导航、列表扫描与选择
```

分享详情的浏览器回退必须固定无头，不读取“显示浏览器”设置，也不访问快手首页做通用登录检查。它只负责：

1. 在导航前注册媒体响应监听器。
2. 打开一次详情页。
3. 从首次响应、DOM `video` 地址或一次播放触发中取得媒体地址。
4. 播放后以 100 ms 时间片泵送响应事件，命中后立即返回，总预算独立限制为 8 秒。
5. 捕获失败时直接结束，不得通过 `reload` 重置页面、Cookie 或媒体状态。
6. 只提交一个资源并关闭会话。

唯一的可见窗口例外是服务端明确要求重新登录：无头详情页识别到登录入口后，可以关闭静默会话并打开一次人工登录窗口；登录状态落盘后必须重新回到无头详情会话。这个窗口由认证状态触发，不由“显示浏览器”设置触发，也不属于启动闪窗。

旧实现先在 HTTP 路径记录一次“检测到分享链接”，失败后又进入通用浏览器路径并再次识别，因此日志看起来像处理了两遍。输入类型应只在顶层分流处确认一次；各解析器只记录当前阶段，不重复宣告输入分类。

## 启动、短链与登录态边界

- 短链展开使用流式响应，连接/读取超时上限分别为 5 秒和 12 秒，整个链路总预算为 15 秒且最多跟随 5 次重定向；公网 DNS 校验也必须计入这 15 秒，并在校验后按剩余时间重算 curl 超时。超时 DNS 校验由进程级有界闸门限制并发，禁止为每个任务无限遗留线程。
- 重定向最终响应必须直接交给详情解析复用，不能为同一详情页再发一次 GET；缓存响应的 4xx/5xx 同样属于 HTTP 快路失败，必须关闭响应并回退浏览器，不能直接终止任务。
- 短链 HTTP 展开失败时仍保留“分享任务”分类，浏览器只能进入单详情捕获流程，不能误落入普通首页/列表扫描。
- 详情 HTML 最多读取 2 MiB，超过预算立即关闭响应并回退浏览器；逐跳公网校验和允许域约束保持不变。
- 有头会话的目标 URL 公网预检必须早于 `chromium.launch()`，避免 DNS/策略检查期间让新窗口长时间停留在 `about:blank`。
- 登录检查已经成功导航到目标页时，后续任务路由必须复用当前页面；首页搜索和人工登录成功后也不得无条件再次跳首页。
- Profile 登录探针使用独立 `APIRequestContext`，只复制目标 URL 可携带的 Cookie，并显式继承浏览器代理。禁止使用 `page.request`，否则响应 `Set-Cookie` 会修改 BrowserContext 的 Cookie Jar。
- 自动写回登录文件只允许发生在服务端明确返回已登录时；探针不确定但 DOM 仍可用时可以继续当前任务，不得覆盖磁盘状态。
- 持久化至少要求主站 `userId`，并把 `kuaishou.server.web_st` 与 `kuaishou.server.webday7_st` 视为同一可轮换认证家族。自动快照丢失既有认证家族时保留原文件，但不能因此把当前已确认会话误判为未登录。
- 人工登录经过连续服务端确认后可以显式替换旧认证家族；保存失败时登录窗口不得宣称已完成跨会话恢复。
- 人工登录轮询使用统一的 2 分钟截止线，单次 Profile 探针不得超过剩余预算；服务端结果不确定且已有旧状态时不能绕过认证家族保护。
- 搜索结果点击只选择点击后新增且已离开 `about:blank` 的页面；无可用新页时检查原页，禁止递归点击同一结果制造 popup。

## 验证方法

### 有头任务窗口验证

加载真实的快手保存态，创建有头 context 并导航快手首页。窗口监控应满足：

- 新增可见 Chrome HWND 数量为 1。
- 同一 HWND 的标题可以经历“无标题”、`about:blank` 和快手页面标题。
- 不出现第一个 HWND 消失后又创建第二个 HWND 的序列。

### localStorage 恢复验证

不能在完整快手页面加载后逐值比较 localStorage，因为站点脚本会立即更新会话时间、实验配置等值，造成假阴性。正确方法是：

1. 在 `https://www.kuaishou.com/` 下注册一个只返回空 HTML 的同源探针路由。
2. 导航该探针页，使 context 初始化脚本运行，但不执行快手业务脚本。
3. 比较保存态和页面 localStorage 的键和值。

探针页必须与保存态 origin 同源；`about:blank` 不具备目标 origin 的 localStorage 空间，不能用来验证。

### 分享任务验证

即使配置中开启“显示浏览器”，分享专用会话也应满足：

- 新增可见 Chrome HWND 数量为 0。
- 顶层“检测到分享/详情链接”日志只有一条。
- 通用 `_run_browser_session()` 不被调用。
- `sig_finished` 只发射一次。
- 单条详情最多提交一个下载资源。

## 常见误区

- 只按窗口标题计数：同一个窗口导航时标题会变化，必须按 HWND 去重。
- 看到 `about:blank` 就认定多开：真正问题是句柄关闭后重建，而不是一个窗口曾显示空白页。
- 把所有模式都改为 Cookie-only：会丢失无头会话可能需要的 IndexedDB。
- 在 `page.goto()` 之后才监听响应：首屏媒体请求可能已经结束，后续只能靠重复刷新碰运气。
- 用共享 `page.request` 校验登录：响应 Cookie 会污染浏览器会话，随后持久化可能把缩水状态覆盖到磁盘。
- 把状态保存失败等同于登录失败：会错误打开新的登录窗口，并制造额外 `about:blank` 与重复导航。
- HTTP 快路失败后进入完整首页扫描：这既增加延迟，也会制造重复日志和不必要的登录窗口。
- 用站点加载后的 localStorage 与磁盘快照做严格比较：站点自身改写会产生假失败，应使用同源空白探针。

## 维护检查清单

修改快手浏览器启动、登录态或分享解析代码时，至少确认：

- context 初始化脚本仍早于首个页面创建。
- 有头会话不把非空 `origins` 直接交给 Playwright。
- 无头会话仍可原生恢复完整 `storage_state`。
- 分享链接在 `run()` 顶层提前返回，不落入普通列表任务。
- 分享浏览器回退显式 `headless=True`。
- 媒体监听器在导航前注册，并在退出时移除。
- 详情捕获失败路径不调用 `reload`，等待按短时间片可中断。
- Profile 探针不使用共享 Cookie Jar，且代理配置与浏览器一致。
- 服务端结果不确定时不自动覆盖登录文件，Cookie 家族轮换测试仍通过。
- 登录成功后的目标页和首页搜索不发生重复导航。
- 短链展开失败（包括缓存详情响应返回 4xx/5xx）仍走分享会话；搜索结果空白 popup 最多等待一次导航，不会被置前或由重叠选择器重复点击。
- 人工登录总等待不超过 2 分钟，探针超时受剩余预算约束。
- 浏览器关闭仍由 Spider 所在线程执行，避免跨线程 Playwright greenlet 错误。
