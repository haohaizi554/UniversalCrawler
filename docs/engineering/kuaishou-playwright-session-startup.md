# 快手 Playwright 会话启动与闪窗排障实践

> 本文记录快手任务从会话启动、短链解析、登录态复用到 CDN 下载交接的诊断方法和当前推荐实现。相关代码主要位于 `app/spiders/kuaishou/`、`app/core/downloaders/kuaishou.py` 和共享下载链路。

更新日期：2026-07-18

## 问题结论

“任务启动时先闪过一个小窗，再出现正式浏览器”并不是快手输入类型、Node.js 控制台或用户误操作导致的。根因是有头模式创建 `BrowserContext` 时，Playwright 为恢复 `storage_state.origins` 中的 localStorage，会执行一段内部导航流程：

1. 创建临时页面或窗口。
2. 导航到保存态中的 origin。
3. 写入 localStorage 等站点存储。
4. 关闭临时页面。
5. 应用代码随后再创建真正的任务页面。

因此，只要本地存在包含 `origins` 的快手登录态，任何使用有头模式的快手任务都可能出现闪窗，与关键词、主页链接、用户链接或分享链接无关。

## 如何确认根因

只观察 UI 很容易把临时窗口误判为登录窗口、反检测脚本或 Chromium 启动器。排查时应同时记录窗口句柄和进程：

- 用 `EnumWindows` 枚举可见顶层窗口。
- 用 `GetWindowThreadProcessId` 取得窗口所属 PID。
- 只记录本次启动后新增的 `chrome.exe` / `chromium.exe` 窗口。
- 以 HWND 为身份，而不是只看窗口标题；标题会从“无标题”变为 `about:blank`，再变为站点标题。

旧实现的典型序列是：同一 Chrome PID 下出现 HWND A，A 很快消失，随后出现 HWND B。修复后的有头任务应始终只有一个 HWND，其标题可以变化，但句柄不应发生“关闭后重建”。

还应先排除 Playwright 驱动进程本身。Windows 版 Playwright 的 `PipeTransport` 已使用隐藏窗口标志启动 Node.js；如果监控到的是 Chrome 顶层 HWND，就不应继续从控制台窗口方向猜测。

## 当前恢复策略

有头会话与无头会话需要采用不同策略。

### 有头会话

有头会话创建 context 时只把 Cookie 交给 Playwright，并将 `origins` 置空。随后在创建第一个真实页面之前，通过 context 级 `add_init_script()` 按 `location.origin` 恢复 localStorage。

这样既保留登录 Cookie 和站点本地状态，又不会触发 Playwright 的内部 origin 恢复页。初始化脚本必须满足以下约束：

- 在 `context.new_page()` 之前注册。
- 只恢复快手和其已授权关联域名，不能把任意 origin 写入浏览器。
- 值按字符串原样恢复，不解析或重新序列化业务内容。
- 无有效 origin 存储时不注入空脚本。

### 无头会话

无头会话没有可见闪窗问题，应继续把完整 `storage_state` 交给 Playwright。原因是未来或当前登录态可能包含 IndexedDB；仅靠 localStorage 初始化脚本无法等价恢复 IndexedDB。

不要为了统一代码路径，让无头会话也退化为“Cookie + localStorage”。这会在表面消除一个不存在的 UI 问题，却可能悄悄破坏静默采集的认证状态。

## 分享链接必须独立分流

分享或单条详情链接与主页、用户页、搜索页不是同一种任务。当前推荐流程是：

```text
规范化输入与短链重定向
  -> 判断是否为分享/详情链接
    -> 纯 HTTP 快速解析
      -> 成功：提交一个下载任务并结束
      -> 失败：进入固定无头的详情页捕获会话
    -> 普通任务：按设置进入登录检查、目标页导航、列表扫描与选择
```

分享详情的浏览器回退必须固定无头，不读取“显示浏览器”设置，也不访问快手首页做通用登录检查。它只负责：

1. 在导航前注册媒体响应监听器。
2. 打开一次详情页。
3. 从首次响应、DOM `video` 地址或一次播放触发中取得媒体地址。
4. 播放后以 100 ms 时间片泵送响应事件，命中后立即返回，总预算独立限制为 8 秒。
5. 捕获失败时直接结束，不得通过 `reload` 重置页面、Cookie 或媒体状态。
6. 只提交一个资源并关闭会话。

唯一的可见窗口例外是服务端明确要求重新登录：无头详情页识别到登录入口后，可以关闭静默会话并打开一次人工登录窗口；登录状态落盘后必须重新回到无头详情会话。这个窗口由认证状态触发，不由“显示浏览器”设置触发，也不属于启动闪窗。

旧实现先在 HTTP 路径记录一次“检测到分享链接”，失败后又进入通用浏览器路径并再次识别，因此日志看起来像处理了两遍。输入类型应只在顶层分流处确认一次；各解析器只记录当前阶段，不重复宣告输入分类。

## 关键词搜索卡顿复盘

普通关键词搜索曾复用“打开首页 -> 定位输入框 -> 输入并回车 -> 等待 SPA 列表”的交互链路。实测中首页与登录检查约耗时 25 秒，搜索列表又吃满 60 秒超时；进入列表扫描后还会再次等待同一批卡片，用户看到的就是长时间停在搜索阶段。

当前实现直接导航快手官方结果路由 `https://www.kuaishou.com/search/video?searchKey=...`，并明确划分等待责任：

1. 登录检查预加载最终搜索结果页，后续路由复用当前页面，不再回首页或重复导航。
2. 导航只等待 `commit`，最大预算 15 秒；它只负责确认主文档开始响应。
3. 视频卡片就绪由列表扫描阶段负责，整段最多等待 30 秒，不能在导航层和扫描层各等一次。
4. 已保存 Cookie 的服务端探针不确定时，先进行 1.5 秒短探针，再给 SPA DOM 最多 5 秒、每 250 ms 可中断复检，避免页面尚在挂载就误判登录失效。

修复后的真实完整阶段约为：登录与目标页建立 9.6 秒、目标路由复用 0 秒、卡片就绪 2.6 秒，总计约 14.7 秒。计时必须分别覆盖“登录检查、目标导航、列表就绪”，只看总耗时无法识别重复等待。

排查期间曾尝试把 `www.kuaishou.com` 固定到一个通过 TCP/TLS/HEAD 探测的 IP。该方案已撤销：单个地址虽然探针成功，Chromium 完整导航仍可能随机出现 `ERR_CONNECTION_CLOSED`，并且固定地址会丢掉浏览器原有的多地址容错。站点浏览器链路应保留 Chromium 的 DoH 与地址回退能力；只有具备逐跳校验和固定契约的受控 HTTP 请求才适合绑定解析结果。

这类 SPA 卡顿的维护原则是：

- 优先使用站点稳定的官方结果路由，避免把键盘交互当作路由 API。
- 每一种“资源就绪”只能有一个等待所有者，调用方不得在返回后重复等待同一条件。
- 登录态“不确定”与“明确未登录”必须分开处理，短暂 DOM 未就绪不能触发人工登录。
- 网络优化必须用完整浏览器导航验证，TCP/TLS/HEAD 成功不能证明 SPA 页面可稳定加载。
- 阶段日志应描述正在等待的真实条件，不能用字面量 `...` 代替状态；列表摘要必须是无前导换行的单行文本，完整正文留在详情面板。

## 启动、短链与登录态边界

- 短链展开使用流式响应，连接/读取超时上限分别为 5 秒和 12 秒，整个链路总预算为 15 秒且最多跟随 5 次重定向；公网 DNS 校验也必须计入这 15 秒，并在校验后按剩余时间重算 curl 超时。超时 DNS 校验由进程级有界闸门限制并发，禁止为每个任务无限遗留线程。
- 重定向最终响应必须直接交给详情解析复用，不能为同一详情页再发一次 GET；缓存响应的 4xx/5xx 同样属于 HTTP 快路失败，必须关闭响应并回退浏览器，不能直接终止任务。
- 短链 HTTP 展开失败时仍保留“分享任务”分类，浏览器只能进入单详情捕获流程，不能误落入普通首页/列表扫描。
- 详情 HTML 最多读取 2 MiB，超过预算立即关闭响应并回退浏览器；逐跳公网校验和允许域约束保持不变。
- 有头会话的目标 URL 公网预检必须早于 `chromium.launch()`，避免 DNS/策略检查期间让新窗口长时间停留在 `about:blank`。
- 登录检查已经成功导航到目标页时，后续任务路由必须复用当前页面；首页搜索和人工登录成功后也不得无条件再次跳首页。
- Profile 登录探针使用独立 `APIRequestContext`，只复制目标 URL 可携带的 Cookie，并显式继承浏览器代理。禁止使用 `page.request`，否则响应 `Set-Cookie` 会修改 BrowserContext 的 Cookie Jar。
- 自动写回登录文件只允许发生在服务端明确返回已登录时；探针不确定但 DOM 仍可用时可以继续当前任务，不得覆盖磁盘状态。
- 持久化至少要求主站 `userId`，并把 `kuaishou.server.web_st` 与 `kuaishou.server.webday7_st` 视为同一可轮换认证家族。自动快照丢失既有认证家族时保留原文件，但不能因此把当前已确认会话误判为未登录。
- 人工登录经过连续服务端确认后可以显式替换旧认证家族；保存失败时登录窗口不得宣称已完成跨会话恢复。
- 人工登录轮询使用统一的 2 分钟截止线，单次 Profile 探针不得超过剩余预算；服务端结果不确定且已有旧状态时不能绕过认证家族保护。
- 搜索结果点击只选择点击后新增且已离开 `about:blank` 的页面；无可用新页时检查原页，禁止递归点击同一结果制造 popup。

## 短链 HTTP 快路、Cookie 与 CDN 交接

分享文案到本地成品不是一次请求，而是具有不同信任边界和生命周期的流水线：

```text
分享文案或 URL
  -> 提取并规范化地址
  -> 短链逐跳校验、DNS 固定和手动重定向
  -> 按当前跳作用域附带保存态 Cookie
  -> 复用最终详情响应，解析 __APOLLO_STATE__.photoUrl
  -> 构建快手下载任务
  -> 按 URL 选择普通 HTTP 或 HLS 下载
  -> HTTP 写入 .downloading 后原子替换；HLS 使用各自受管临时工作区
  -> 下载队列空闲后执行一次本地媒体对账扫描
```

### 短链逐跳传输边界

- 只允许 `kuaishou.com`、`chenzhongtech.com` 的精确主机或真实子域；不能对完整 URL 做字符串包含判断。
- 每一跳都先规范化为 ASCII/IDNA 传输 URL，再解析并校验公网 IPv4/IPv6 地址；自定义端口必须原样进入 `CurlOpt.RESOLVE`。
- curl 请求显式设置 `allow_redirects=False`。收到重定向后关闭当前响应，重新校验下一跳并生成新的 DNS 固定项，不能沿用上一跳结果。
- 整条短链最多跟随 5 次重定向，总预算 15 秒，详情 HTML 上限 2 MiB；DNS 校验、连接和正文读取都消耗同一截止时间。
- 短链固定 DNS 与显式代理不能同时保证目标连接地址。检测到代理、无法构造 `CurlOpt.RESOLVE` 或策略能力缺失时必须失败关闭 HTTP 快路，转入受保护的无头详情会话。
- 直连快路显式设置 `CurlOpt.PROXY=""`，避免 curl 从环境变量继承代理后绕开已验证的地址固定。

这些约束只授权当前快手详情跳转链，不授权最终媒体 CDN。CDN 地址必须作为新的外部资源重新进入下载层的公网策略。

### 保存态 Cookie 的最小作用域

- 分享任务只读取一次磁盘登录态快照，之后把它当作只读输入；不能在逐跳请求中反复读盘或写回。
- 每个请求跳转都按目标 URL 重新筛选 Cookie，严格执行 domain、path、secure 和 expiry 规则；不允许把上一跳 Cookie 字典直接复制到下一跳。
- `kuaishou.com` 与 `chenzhongtech.com` 之间也不能凭“同属快手”共享不匹配作用域的 Cookie。
- 缺少 domain/path 等作用域、过期时间非法或 Cookie 结构损坏时，当前跳按匿名请求处理；不得放宽规则、修补为全域 Cookie，或把 Cookie/完整 storage state 写入日志。
- 只有短链 HTTP 快路消费这份保存态；直接输入完整详情 URL 时，当前 HTTP 解析仍是匿名请求。`photoUrl` 指向的签名 CDN URL 不继承详情 Cookie；任务只传递下载所需的 User-Agent 与 Referer，除非后续有明确且经过作用域验证的 CDN Cookie 契约。

匿名详情 HTML 可能只包含骨架状态，而带作用域登录态的响应才包含可用 `photoUrl`。因此“匿名快路未命中”不等于作品不存在；它应回退固定无头详情会话，而不是重进主页、刷新 Cookie 或重复导航。

### Apollo 详情与 CDN 地址交接

- 短链最终响应与详情解析是同一份响应，必须直接复用，禁止为同一详情页再发一次 GET。
- 当前解析入口从 `window.__APOLLO_STATE__` 的 `VisionVideoDetailPhoto:<id>` 读取标题和 `photoUrl`；缺字段、响应错误或超出正文预算都应关闭响应并回退浏览器。
- `photoUrl` 是不透明的带签名 CDN 地址。路径、查询参数和参数顺序不得被重新编码、删减或拼接；不得把详情主机的 DNS 固定项、Host 头或 Cookie 套到 CDN 主机。
- 下载任务保留 User-Agent、详情 Referer、trace 信息和显式策略。地址字符串包含 `.m3u8` 时标记为 HLS 首选，其余标记为普通 HTTP 首选；首选策略不可用或失败时，统一策略链仍可继续回退。
- `emit_video()` 会给外部媒体任务写入 `_network_policy=public`。在此前提下，普通 HTTP 下载会在请求前校验公网 URL，并逐响应检查重定向；携带同一公网策略的 HLS `curl_cffi` 兜底还会对每一跳使用 `CurlOpt.RESOLVE` 固定已验证地址。外部 N_m3u8 工具等其他 HLS 路径不经过这个 helper，不能把该保证泛化到整个策略链。
- 普通 HTTP 下载先写入 `*.downloading`，成功后原子替换为最终文件，中断、重试和续传只操作临时文件。HLS 与外部工具使用各自受管的临时工作区和收尾契约，不能假定都采用 `.downloading`。

### 下载时间流与媒体目录扫描

采集完成、下载入队、CDN 传输完成和媒体目录对账是四个独立时刻。日志中先出现“爬虫任务结束”，随后才出现“下载任务完成”是正常的异步流水线，不表示任务被执行两次；秒级日志时间也不能用于推断亚秒级线程先后。

目录监控每 0.8 秒只读取目录元数据，但旧实现检测到 `.downloading` 创建、下载中间文件变化和最终原子替换时，都会立即触发一次完整目录遍历。180 ms 去抖只能合并同一小段突发，无法覆盖持续十几秒的下载，所以会出现“开始扫描/扫描完成”在下载期间反复穿插的时间流。

当前 GUI 与 WebUI 都会在观察到下载活跃时延迟新扫描，但两端的扫描执行协调方式不同：

1. 目录变化时先查询下载管理器的 active、dispatching 和 queued 数量。
2. 当前快照仍有下载工作时只记录一个 deferred/dirty 状态，不新启动完整扫描。
3. 活跃期用单一轻量定时重检等待队列空闲；连续文件变化不会创建多条重检链。
4. 取得空闲快照后清除 dirty 状态并发起一轮完整扫描，纳入最终成品。
5. 应用空闲时由 Explorer 或其他程序造成的目录变化仍按原去抖策略及时扫描。
6. Web 控制器停止时取消重检句柄和扫描任务；GUI 的单次 Qt 回调不可取消，依靠 closed/deferred 状态让旧回调失效。目录切换后的旧 generation 扫描结果不得提交到新目录。

`pending_work_counts()` 是时点快照，不是扫描与下载之间的全局互斥锁。如果扫描已经启动后才有任务入队，既有遍历不会被强制中止。Web 用 `_external_media_scan_task` 和 pending 标志串行合并运行中的扫描；GUI 在空闲路径使用 180 ms 去抖，并用取消 token/generation 阻止旧结果提交，但后续外部变化仍可能排入新一轮遍历。

因此截图所示“下载已活跃且临时文件持续变化”的脏窗口，预期时间流会收敛为“若干目录变化被延期 -> 下载完成 -> 发起一轮合并扫描”，而不是每个临时文件变化都立即遍历媒体库。扫描开始前后的新入队任务或新外部变更仍按上述快照与各端协调规则处理。

### 可观测性与保密要求

- 采集、分享解析和下载阶段应分别记录清晰的阶段名。当前采集 trace 与分享下载 trace 可能不同，不能假设 trace ID 天然贯通，也不能只靠同一秒的显示时间猜测先后关系。
- 后续完善可观测性时，应补充短链跳数、回退原因、CDN hostname、传输字节数、速度和分段耗时。
- 新增或修改日志时不得记录 Cookie、Authorization、完整 storage state，或带完整签名查询串的 CDN URL；只记录必要的主机和脱敏路径摘要。当前部分旧 debug 日志仍包含完整 `source_url`，这是待收敛的已知缺口，不能把本条描述成已经全面达成的保密保证。
- 判断 CDN 慢之前，先区分短链/DNS 耗时、浏览器回退、下载排队、真实传输和媒体扫描五段，不能把 UI 中最后出现的阶段当成全部耗时来源。

## 验证方法

### 有头任务窗口验证

加载真实的快手保存态，创建有头 context 并导航快手首页。窗口监控应满足：

- 新增可见 Chrome HWND 数量为 1。
- 同一 HWND 的标题可以经历“无标题”、`about:blank` 和快手页面标题。
- 不出现第一个 HWND 消失后又创建第二个 HWND 的序列。

### localStorage 恢复验证

不能在完整快手页面加载后逐值比较 localStorage，因为站点脚本会立即更新会话时间、实验配置等值，造成假阴性。正确方法是：

1. 在 `https://www.kuaishou.com/` 下注册一个只返回空 HTML 的同源探针路由。
2. 导航该探针页，使 context 初始化脚本运行，但不执行快手业务脚本。
3. 比较保存态和页面 localStorage 的键和值。

探针页必须与保存态 origin 同源；`about:blank` 不具备目标 origin 的 localStorage 空间，不能用来验证。

### 分享任务验证

即使配置中开启“显示浏览器”，分享专用会话也应满足：

- 新增可见 Chrome HWND 数量为 0。
- 顶层“检测到分享/详情链接”日志只有一条。
- 通用 `_run_browser_session()` 不被调用。
- `sig_finished` 只发射一次。
- 单条详情最多提交一个下载资源。

## 常见误区

- 只按窗口标题计数：同一个窗口导航时标题会变化，必须按 HWND 去重。
- 看到 `about:blank` 就认定多开：真正问题是句柄关闭后重建，而不是一个窗口曾显示空白页。
- 把所有模式都改为 Cookie-only：会丢失无头会话可能需要的 IndexedDB。
- 在 `page.goto()` 之后才监听响应：首屏媒体请求可能已经结束，后续只能靠重复刷新碰运气。
- 用共享 `page.request` 校验登录：响应 Cookie 会污染浏览器会话，随后持久化可能把缩水状态覆盖到磁盘。
- 把状态保存失败等同于登录失败：会错误打开新的登录窗口，并制造额外 `about:blank` 与重复导航。
- HTTP 快路失败后进入完整首页扫描：这既增加延迟，也会制造重复日志和不必要的登录窗口。
- 用站点加载后的 localStorage 与磁盘快照做严格比较：站点自身改写会产生假失败，应使用同源空白探针。

## 维护检查清单

修改快手浏览器启动、登录态或分享解析代码时，至少确认：

- context 初始化脚本仍早于首个页面创建。
- 有头会话不把非空 `origins` 直接交给 Playwright。
- 无头会话仍可原生恢复完整 `storage_state`。
- 分享链接在 `run()` 顶层提前返回，不落入普通列表任务。
- 分享浏览器回退显式 `headless=True`。
- 媒体监听器在导航前注册，并在退出时移除。
- 详情捕获失败路径不调用 `reload`，等待按短时间片可中断。
- Profile 探针不使用共享 Cookie Jar，且代理配置与浏览器一致。
- 服务端结果不确定时不自动覆盖登录文件，Cookie 家族轮换测试仍通过。
- 登录成功后的目标页和首页搜索不发生重复导航。
- 短链展开失败（包括缓存详情响应返回 4xx/5xx）仍走分享会话；搜索结果空白 popup 最多等待一次导航，不会被置前或由重叠选择器重复点击。
- 人工登录总等待不超过 2 分钟，探针超时受剩余预算约束。
- 浏览器关闭仍由 Spider 所在线程执行，避免跨线程 Playwright greenlet 错误。
- 短链每一跳都重新校验主机、公网地址、DNS 固定项和 Cookie 作用域，代理或固定能力缺失时失败关闭。
- 详情响应只消费一次，CDN URL 原样交接；普通 HTTP 与 HLS 的公网校验能力描述准确。
- GUI/WebUI 观察到下载工作时不新启完整媒体扫描，并在后续空闲快照发起一轮合并扫描；已启动扫描和后续新变更仍遵守各端既有协调契约。
