# 配置中心下拉框右侧空白复盘

> 当前执行规则已整理到 [设置、下拉框与国际化契约](../engineering/settings-ui-contract.md)。本文保留事故背景、根因和经验教训。

## 现象

配置中心的平台设置页中，窄列下拉框（例如“超时”）打开或聚焦时，控件右侧会出现明显空白。用户看到的是文本被挤在左侧，而右侧留出一块没有内容的白色区域。

## 根本原因

项目已经在 `app/ui/components/combo_popup.py` 中统一了下拉框策略：隐藏 Qt 原生下拉箭头区域，由统一的 popup delegate 和几何锁定负责列表绘制。

但配置中心页面自己的 `SettingsCombo` QSS 又重新保留了原生下拉区域：

- 控件右 padding 使用了 `30px`。
- `QComboBox::drop-down` 宽度设置为 `28px`。
- 箭头本身没有有效视觉内容，导致窄列中这部分只表现为空白。

在宽控件里这个问题不明显；在平台设置的“超时”这种 96px 左右的窄列里，保留区会吃掉接近三分之一的宽度，所以空白非常刺眼。

## 修复方式

- `SettingsCombo` 的右 padding 收紧到 `10px`。
- `QComboBox#SettingsCombo::drop-down` 改为 `width: 0px`。
- `QComboBox#SettingsCombo::down-arrow` 显式 `image: none` 且宽高为 `0px`。
- 增加窄列“超时”下拉框契约测试，断言：
  - edit field 不再被原生箭头区域大量挤压；
  - popup 宽度锁定在控件宽度内；
  - 选中行背景能绘制到右边缘；
  - 被人为放宽的 popup 会被重新收回目标宽度。

## 二次问题：推荐文案被截断

原生箭头空白修掉以后，平台设置里的“超时”列仍然沿用旧的固定列宽。旧宽度只适合 `60` 或 `60 秒` 这类短文本；配置选项统一改为 `60 秒（推荐）` 后，`timeout` 列仍可能被压缩到旧预算，结果控件右侧没有空白 gutter 了，但文字本身被裁掉。

这类问题的根因不是 popup，而是列宽预算没有跟随 option label 变化。修复时不能只写死一个更大的数字，还要让 GUI 依据实际下拉选项测量最长 label，并把结果纳入平台设置表格的列宽计算；WebUI 也要同步调整对应 grid 列宽。

这次还暴露出一个配套问题：平台表格行有 5 个固定单元，至少存在 4 个水平间距，但旧预算只扣了 30px。列宽计算少扣间距时，右侧列会被末端挤压；修复时必须把单元格 margin 和 gap 都纳入总宽预算。

新增契约测试必须同时覆盖：

- 当前默认项包含“推荐”时，控件宽度足够；
- `SC_ComboBoxEditField` 的可用文本宽度大于等于最长选项文本宽度；
- WebUI 平台设置 grid 的 `timeout` 列不再使用旧的 `96px`。
- 平台行固定列宽总和不得超过扣除 margin 和 gap 后的真实内容预算。

## 后续规则

- GUI 下拉框的 popup 行为必须集中走 `app.ui.components.combo_popup`，页面级 QSS 不应重新引入原生 `drop-down` 保留宽度。
- 对窄列 combo（平台数量、超时、分页、顶部数量）必须有针对性截图或契约测试；只测宽控件会漏掉这类问题。
- 如果需要显示下拉提示图标，应使用显式图标控件或自绘方案，不要依赖 Qt 原生 arrow 区域。
- 所有设置页 combo 的 `value` 和 `label` 要分离；推荐文案只能作为 label，不能污染提交给后端的 value。
- 所有写在表格列里的 combo，列宽必须按最长 label 做预算；默认/推荐文案变化时，要同步检查 GUI 和 WebUI。
- 表格/网格列宽计算必须显式扣除外边距和列间距；不要把“看起来还有空间”的 stretch 当作可用内容宽。
