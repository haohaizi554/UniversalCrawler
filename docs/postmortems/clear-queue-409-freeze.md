# 清空队列 409 / UI 卡死复盘

## 事故现象

删除所有排队下载任务的 GUI 操作可能会导致窗口卡死，并且在桌面端运行环境中，最终触发类似 409 的应用失败。

可见现象是：用户点击“清空全部 / 删除全部”控制按钮后，UI 停止响应，进程没有恢复，而是直接退出。

## 根本原因

清空队列流程把一个高频、大批量的 UI 操作，当成了一个小规模的同步编辑来处理：

- 它通过前端快照路径收集队列行数据，而这个路径可能会扫描并复制一个很大的 video map。
- 它通过“替换式”的状态更新来删除大量条目，而不是使用原地批量删除。
- 删除完成后，它触发了一次完整的强制前端状态刷新。
- 某个 host adapter 路径缺少 `refresh_frontend_state()`，导致 UI 回调在恢复过程中可能抛出异常。
- 本应放到 Qt 主线程之外执行的工作，在大量队列变更时可能运行在主线程上。

## 修复模式

修复后的流程遵循一个规则：所有大型 UI 变更都应该这样处理：

1. 只通过窄状态适配器方法收集 ID，例如 `queue_item_ids()`。
2. 将昂贵的删除工作放到 Qt 主线程之外执行。
3. 如果 manager 支持批量取消，则一次性批量取消排队下载任务。
4. 通过 `AppState.remove_videos(..., publish=False)` 原地删除 videos。
5. 等状态变更完成后，只发布一个关键批量事件：`videos.remove_many`。
6. 请求基于 topic 的前端刷新，而不是强制完整快照刷新。

## 后续规则

- 不要使用完整前端快照来支撑破坏性的批量 UI 操作。
- 不要把 `replace_videos()` 当成删除大量条目的快捷方式。
- 批量操作时，不要为每一个被删除的条目发布一个事件。
- 不要从后台线程直接调用 UI 刷新方法；应该把最终的回调封送回 Qt 线程。
- 任何高频、大批量队列操作，都需要至少包含 10,000 个条目的回归测试。
- controller mixin 使用的任何新 host adapter 方法，都必须同时存在于 GUI adapter 和 test adapter 中。

## 回归测试覆盖

回归测试应该持续断言 `clear-all` 操作满足以下要求：

- 使用 `queue_item_ids()`，而不是 `get_snapshot()`。
- 在可用时调用批量取消。
- 删除 videos 时不替换整个 video 字典。
- 只发出一个 `videos.remove_many` 事件。
- 只刷新受影响的前端 topics。
