# 发布构建面板与统一版本契约设计

## 2026-07-20 面板布局与三模式收束

本节是对后文“发布模式”和“发布面板”的增量修订。发生冲突时，以本节为准。

### 用户可见的三种模式

底层 `ReleaseMode` 保持兼容，不修改命令行、Runner 和既有结果协议。GUI 增加一层
用户意图映射，只展示三种可理解的模式：

| GUI 模式 | 可用条件 | 底层模式 | 远端写入 |
| --- | --- | --- | --- |
| 同版本修复发布 | 目标版本等于远端版本 | `SAME_RELEASE_REPAIR` | 允许 |
| 高版本正式发布 | 目标版本高于远端版本 | `NEW_RELEASE` | 允许 |
| 本地构建 | 任意版本；低版本或远端未知时自动进入 | `LOCAL_DEBUG`、`LOCAL_REBUILD` 或 `OFFLINE_DEBUG` | 严格禁止 |

版本关系首次确定或目标版本发生有效变化时，面板选择对应的推荐模式。操作者可以随时
显式切换到“本地构建”；同版本修复和高版本发布按钮在版本关系不匹配时不可用。
远端版本未知时只允许本地构建，并把请求映射为 `offline_debug=True`，不得猜测远端
发布模式。

模式选择与版本关系分离：目标版本决定哪些发布模式合法，操作者意图决定本次是否
真的发布。选择本地构建后，即使目标版本高于远端，也不得产生 Git、Release、上传
或签名动作。

### 模式默认值

每种模式只在“首次进入该模式”时应用默认值。随后表单刷新、远端查询完成、代理变化
或普通文本编辑都不得覆盖操作者已经手动调整的复选框。

三种模式共享以下默认构建项：

- 应用目标版本。
- 构建便携版。
- 构建安装包。
- 运行冒烟测试。

“本地构建”关闭并禁用所有签名及远端写入项。

“同版本修复发布”额外默认启用：

- 签署更新清单。
- 创建或复用现有标签。
- 创建或更新 Release。
- 上传发布资产。
- 上传生产公钥。
- 校验远端资产。

同版本修复不默认提交版本变更或推送 `main`，因为版本事实源没有发生变化。

“高版本正式发布”额外默认启用：

- 签署更新清单。
- 提交版本变更并推送 `main`。
- 创建或复用标签。
- 创建或更新 Release。
- 上传发布资产。
- 上传生产公钥。
- 校验远端资产。

“生成更新清单密钥”和“轮换信任锚”在所有模式下都保持未勾选。这两项会改变更新
信任链，不能因模式切换自动开启。远端写入开始前仍必须显示最终确认摘要。

### 双栏单页布局

面板按已确认的参考图采用双栏工作台，不再把六个区域纵向堆叠：

```text
+--------------------------------+--------------------------------------+
| 1 版本信息与模式               | 6 执行进度与日志                    |
| 2 构建选项                     |   模式 / 状态 / 开始 / 取消         |
| 3 签名与信任                   |   校验信息                          |
| 4 代码仓库与发布               |   进度条                            |
| 5 网络与代理                   |   日志工具栏                        |
|                                |   可扩展日志面板                    |
+--------------------------------+--------------------------------------+
```

- 左栏负责配置，右栏负责执行反馈；两栏之间不交叉拥有控件。
- 移除整页 `QScrollArea`，正常桌面窗口内不显示外层滚动条。
- 左侧卡片使用一致的边框、内边距、标题编号和控件高度。
- 构建选项使用紧凑可点击选项卡；其余选项使用项目复选框、主题下拉框和标准输入框。
- 右侧日志面板占用剩余高度，空态、运行态、失败态和完成态复用同一容器，不重建整页。
- 窗口默认尺寸面向 1366×768 及以上桌面；空间不足时压缩卡片间距和输入宽度，不引入
  外层滚动条，也不允许控件越界或文字被裁切。
- 样式只消费项目主题 token、`WindowChromeFrame` 和现有主题控件，不复制独立配色。

### 状态与数据流

模式策略由纯函数根据“目标版本、远端版本、用户选择”生成 UI 投影和默认选项；
`ReleaseBuilderWindow` 只把投影批量应用到控件。构建时仍由现有
`BuildRequest -> validate_build_request -> ReleaseProcessController` 链路执行，
不新增第二套发布实现。

模式切换必须使用信号阻断批量更新控件，完成后只触发一次校验，避免复选框逐项切换
造成重绘、重复远端查询或中间非法状态闪烁。

### 验收

- 三种 GUI 模式的可用条件、默认勾选和请求映射均有纯逻辑单元测试。
- 本地模式无论目标版本为何，生成的 `BuildRequest` 都不包含远端写入或签名动作。
- 同版本和高版本发布默认值分别符合上表，危险密钥轮换始终关闭。
- 参考尺寸下左右卡片边界、日志高度和工具栏不重叠，且不存在外层滚动条。
- 浅色、深色和项目强调色下均使用主题 token。
- 远端发布仍经过确认弹窗，发布工具完整测试集全绿。

## 背景

当前发布链路已经能够完成便携包、安装包、更新清单、Ed25519 签名和 GitHub
Release 资产生成，但主要由命令行驱动。完整发布还需要操作者在多个命令之间手工
传递版本、代理、发布说明、密钥和资产路径，过程缺少统一的状态展示、进度反馈与
失败恢复入口。

项目版本也尚未真正做到单一事实源。运行时已有 `shared/version.py`，但
`pyproject.toml`、Inno Setup 默认值、FastAPI 元数据、Web 静态兜底、CI 断言和
当前版本文档仍可能保存独立版本值。发布前只修改其中一处会造成 GUI、WebUI、
Python 包、安装器和更新清单显示不同版本。

本设计把 `packaging/build_release.py` 保持为发布构建的唯一入口，在不破坏现有
无界面调用和测试的前提下，为人工构建提供工业化 Qt 面板，并同时收束产品版本
契约。

## 目标

1. 直接运行 `packaging/build_release.py` 时打开主题化发布构建面板。
2. 构建、签名、Git 和网络上传运行在隔离子进程，Qt UI 线程只处理显示、输入和
   有界事件消费。
3. 日志实时显示在面板中，同时写入经过脱敏的持久化日志文件。
4. 目标版本低于远端最新版时进入本地调试模式；高于远端最新版时进入新版本发布
   模式；等于远端最新版时默认本地重构，可显式选择修复同版本 Release。
5. 发布面板中的版本切换真实更新项目版本，而不是只覆盖安装包文件名。
6. `shared/version.py` 成为产品版本的唯一事实源，其他运行时位置全部读取它。
7. 本地调试安装包继续携带项目当前生产公钥，能够验证之后发布的远端热更新；
   本地调试不生成临时密钥、不签署本地更新清单，也不把私钥放入安装包。
8. 远端发布选项按三种 GUI 模式应用安全默认值，危险的密钥生成与信任锚轮换始终
   需要显式选择。
9. 代理选项复用项目已有系统代理、直连、Clash、Clash Verge、v2rayN、
   V2Ray/Qv2ray、sing-box、NekoRay 和自定义端点契约。
10. `bag_15483236.png` 转换为多尺寸标准 ICO，作为发布构建面板图标，不替换
    Universal Crawler Pro 主程序现有品牌图标。

## 非目标

- 不重写 PyInstaller、Inno Setup、更新清单或 Ed25519 签名算法。
- 不把 GitHub token、私钥内容或其他凭据写入项目配置。
- 不为普通用户安装发布构建面板；它是源码仓库中的维护者工具。
- 不自动改写历史 Release 文档、测试夹具或故障复盘中的旧版本号。
- 不用定时器伪造构建进度。
- 不让 GUI 成为另一套发布实现；命令行与面板必须调用同一工作流。

## 总体架构

采用“Qt 控制面板 + 隔离构建子进程 + 带前缀的 JSONL 事件流”。

```text
packaging/build_release.py
        |
        +-- 无参数脚本启动 --> ReleaseBuilderWindow
        |                         |
        |                         +-- 版本/远端预检子进程
        |                         +-- 正式构建子进程
        |                         +-- 面板日志、进度、取消
        |
        +-- --headless/程序调用 --> 现有 release pipeline
                                      |
                                      +-- portable
                                      +-- installer
                                      +-- manifest/signature
                                      +-- git/release/upload/verify
```

`build_release.py` 继续拥有入口和发布工作流装配权，但新增 UI、事件协议、模式判断、
版本写入等职责必须放入独立小模块，避免继续扩大已有脚本：

- `packaging/release_tool/versioning.py`
  - 读取、校验和原子更新项目版本。
  - 同步允许列表中的文档投影。
- `packaging/release_tool/modes.py`
  - 纯函数解析本地调试、同版本重构和新版本发布模式。
- `packaging/release_tool/models.py`
  - 定义不可变构建请求、预检结果、阶段和最终结果。
- `packaging/release_tool/events.py`
  - JSON-safe 事件编码、解析、脱敏和进度阶段定义。
- `packaging/release_tool/runner.py`
  - 把请求映射到现有构建、签名、Git 和上传函数。
- `packaging/release_tool/panel.py`
  - Qt 窗口、表单、状态机、日志视图和 `QProcess` 生命周期。
- `packaging/release_tool/icon_builder.py`
  - 用项目已有 Qt 图像能力生成多尺寸 ICO，不引入新运行时依赖。

模块名是内部契约，不新增对外 Python API。`packaging/build_release.py` 仍是维护者和
CI 使用的唯一公开脚本入口。

## 统一版本契约

### 唯一事实源

`shared/version.py` 中的 `__version__` 是唯一可编辑的产品版本值。

`pyproject.toml` 改为 PEP 621 动态版本：

```toml
[project]
dynamic = ["version"]

[tool.setuptools.dynamic]
version = {attr = "shared.version.__version__"}
```

由此，wheel 元数据、CLI、SDK、GUI 和 Web 后端共享同一个值。其余运行时消费者
必须直接导入 `shared.version.__version__`：

- `packaging/project_meta.py`
- `app/web/server.py`
- GUI 状态栏与更新检查入口
- Web `/api/health`、状态快照与首帧 HTML 注入
- CLI `--version`
- 构建产物名和更新清单

`packaging/installer.iss` 不保留可过期的版本回退值。安装器构建必须由
`build_installer.py` 通过 `/DAppVersion=<version>` 传入版本；缺少该参数时直接
失败，不能静默生成旧版本安装包。

Web 静态文件不再写死 `v3.6.x`。首帧使用服务端注入的版本，尚未获得状态时显示
无版本占位，不得回退到旧常量。

### 版本投影

README 徽章、README 当前版本说明、`docs/README.md` 和 CLI Skill 的当前发布版本
无法在展示时导入 Python，因此它们属于“版本投影”，不是事实源。

版本管理器只同步明确允许的活跃文档字段，并通过测试验证其值等于
`shared.version.__version__`。历史 Release、复盘、架构快照和测试用例中的版本
样例不参与全局替换。

### 面板写入语义

目标版本输入框中的内容是草稿，键入时不修改文件。

以下动作会真实修改项目版本：

1. 操作者点击“应用版本”。
2. 操作者启动构建，而草稿版本与当前项目版本不同，并确认同步。

版本管理器先校验规范化 SemVer，再使用同目录临时文件和原子替换更新
`shared/version.py`，随后同步版本投影并重新读取所有派生值。任一投影更新失败时，
本次同步整体失败，并恢复修改前内容。

成功应用的版本属于明确的工作区变更。后续构建失败不会偷偷回滚版本；面板会列出
已修改文件和当前 Git 状态，让操作者决定继续修复或手工回退。

构建预检必须确认：

- `shared.version.__version__` 等于目标版本。
- `packaging.project_meta.PACKAGE_VERSION` 等于目标版本。
- 构建命令传给 Inno Setup 的版本等于目标版本。
- 生成的安装包名、更新清单版本和预期标签一致。

## 发布模式

模式由规范化目标版本与远端最新稳定 Release 版本比较得出：

| 比较结果 | 默认模式 | 行为 |
| --- | --- | --- |
| 目标版本 `<` 远端版本 | 本地调试 | 构建便携包/安装包；禁用标签、签名和上传 |
| 目标版本 `=` 远端版本 | 本地重构 | 默认同本地调试；可显式开启“修复同版本 Release” |
| 目标版本 `>` 远端版本 | 新版本构建 | 解锁签名、标签、推送、Release 和上传选项 |

远端版本查询失败时不猜测模式。面板显示“远端状态未知”，禁用远端发布操作；操作者
可显式选择离线本地调试。不能用“查询失败”绕过发布安全检查。

“新版本构建”不等于已经发布。只有签名、Release 上传和远端回读验证全部成功后，
最终状态才是“发布成功”；未选择上传时显示“新版本本地构建完成”。

## 本地调试与生产公钥

本地调试只构建应用，不创建 `latest.json`、`latest.json.sig` 或本地临时公钥。

安装包继续打入仓库中已经审计的生产 Ed25519 公钥信任锚。这样，版本较低的本地
调试安装包之后仍能下载并验证由对应生产私钥签署的远端更新。

以下材料不得进入安装包、便携包、Git 仓库、事件流或日志：

- Ed25519 私钥内容。
- GitHub access token。
- `git credential fill` 返回的密码字段。
- 代理认证密码。

正式签名只接收仓库外部私钥路径。私钥文件由签名子进程按需读取，不复制到构建
目录。面板日志最多显示脱敏后的文件名和父目录提示。

## 发布面板

### 窗口框架

窗口复用项目现有：

- `WindowChromeFrame`
- `FramelessWindowChromeController`
- 自定义标题栏和项目主题应用函数
- `theme_colors()` 语义色

窗口保留 Windows 原生缩放、最大化、Snap Layout、DPI 和任务栏行为。面板不写
独立硬编码颜色，不复制一套 QSS。

`bag_15483236.png` 移入发布工具资产目录并生成包含 16、20、24、32、40、48、
64、128 和 256 像素图层的透明 ICO。面板标题栏和任务栏使用该图标；主程序图标
保持不变。

### 信息结构

面板采用单页工作台，不使用多层向导：

1. **版本状态**
   - 当前项目版本。
   - 远端最新版本。
   - 目标版本输入与“应用版本”按钮。
   - 自动解析的模式和风险状态。
2. **构建输入**
   - Release 文档路径，默认 `docs/releases/v<version>.md`。
   - 输出根目录。
   - 便携包、安装包和焦点测试选项。
3. **签名与密钥**
   - 使用外部私钥签署更新清单。
   - 私钥路径选择。
   - 在仓库外生成新密钥对。
   - 显式轮换内置公钥信任锚。
4. **Git 与 Release**
   - 提交版本变更。
   - 推送 `main`。
   - 创建或复用标签。
   - 更新 Release 正文。
   - 上传安装包、清单、签名和公钥。
   - 远端资产回读与签名验证。
5. **网络**
   - 复用项目代理预设。
   - 自定义 HTTP/SOCKS 端点。
   - 当前代理连通性预检。
6. **执行区**
   - 预检、开始、取消和打开输出目录。
   - 总进度、当前阶段和阶段说明。
   - 面板日志、复制、清空视图和导出日志。

远端写操作、签名、标签和上传按本文件顶部“三模式收束”规则设置默认值，并继续使用
以下约束：

- 选择上传更新资产必须同时选择生成签名清单。
- 选择签名必须提供可读的仓库外私钥。
- 选择上传必须先通过远端仓库和代理预检。
- 选择轮换信任锚必须生成或选择新密钥，并重新构建安装包。
- 本地调试模式不允许勾选远端写操作。

开始远端写操作前显示一次摘要确认，列出版本、标签、仓库、代理、Release 文档和
待上传资产，不显示秘密。

## 子进程与事件协议

Qt 使用 `QProcess` 启动同一个 `build_release.py` 的无界面子命令。现有
`main(argv)` 继续保持无界面语义，已有测试调用 `main([])` 不受影响。

脚本入口行为为：

- 作为脚本无参数启动：打开发布面板。
- `--gui`：强制打开面板。
- `--headless` 或现有构建参数：执行无界面工作流。
- Python 测试直接调用 `main(argv)`：执行原有无界面逻辑。

结构化事件使用固定前缀加单行 JSON：

```text
@@UCRAWL_RELEASE_EVENT@@{"kind":"stage","stage":"installer","progress":55}
```

没有该前缀的 stdout/stderr 行作为普通日志展示。事件至少包括：

- `preflight`
- `stage`
- `progress`
- `log`
- `artifact`
- `warning`
- `error`
- `result`

每个事件包含单调递增序号、UTC 时间、阶段和可选字段。事件解析失败不会使面板
崩溃；原始行按警告日志显示。

进度由真实阶段完成事件驱动，不按时间推算。跳过的阶段发出 `skipped`，总进度
保持单调。子进程退出码和最终 `result` 必须同时成功，面板才显示完成。

## 日志与背压

日志必须在面板中实时显示。`QPlainTextEdit` 设置最大 block 数，避免长时间构建
无限增长；事件接收使用有界缓冲，连续普通输出批量追加，错误、阶段变化和最终结果
立即展示。

完整脱敏日志同时写入：

```text
dist/release-logs/<timestamp>-<mode>-<version>.log
```

面板提供复制选中内容、导出日志和打开日志所在目录。点击“清空”只清空视图，不删除
持久化日志。

日志脱敏覆盖：

- GitHub token 和 credential helper 密码。
- 私钥 PEM 内容。
- URL userinfo 和代理认证字段。
- 请求头中的 Authorization、Cookie 和签名秘密。

## 代理契约

代理下拉框直接消费项目已有代理选项，不在发布工具复制名称、端口或 URL。

规范化后的代理通过子进程环境传递：

- `HTTP_PROXY`
- `HTTPS_PROXY`
- `ALL_PROXY`
- 对应小写变量

代理环境同时作用于 GitHub API、`gh`、Git 和资产上传。选择“系统代理”时继承当前
进程环境；选择“直连”时明确移除代理变量；自定义端点必须通过现有 URL 规范化与
校验。

## 执行状态与取消

面板状态机为：

```text
idle
  -> checking_remote
  -> preflight
  -> version_sync
  -> building_portable
  -> building_installer
  -> signing
  -> git
  -> uploading
  -> verifying
  -> succeeded | failed | cancelled
```

执行期间锁定会改变请求语义的表单字段。取消先向子进程发送正常终止请求，并允许
当前原子文件操作收尾；超过限定时间后终止进程树。取消不能把部分资产报告为成功。

所有构建阶段写入临时位置，验证完成后再发布到最终输出目录。正式上传失败时保留
本地已验证资产和日志，允许操作者修复网络后重新执行上传阶段；不得重新签署已验证
且摘要一致的资产。

## 错误处理

预检错误在执行前阻止构建，并定位到具体输入控件。运行期错误包含：

- 用户可读摘要。
- 失败阶段。
- 子进程退出码。
- 脱敏日志路径。
- 已产生且验证通过的资产列表。
- 可重试动作。

远端上传采用幂等语义：重复上传前比较资产名、大小和摘要；已一致则跳过，不一致时
只有明确选择替换才执行覆盖。

## 测试策略

### 版本契约

- `shared.version.__version__` 是合法 SemVer。
- 构建元数据解析值等于唯一事实源。
- wheel 元数据、安装器参数、FastAPI 和前端状态版本一致。
- 原子版本更新成功并同步允许列表中的投影。
- 任一投影写入失败时全部恢复。
- 架构守卫禁止生产目录出现新的产品版本硬编码。
- 历史文档和测试版本夹具不被同步器修改。

### 模式与安全

- 目标版本低于、等于和高于远端版本的模式解析。
- 远端查询失败时仅允许显式离线调试。
- 本地调试请求不能包含签名、标签和上传步骤。
- 上传依赖签名清单和外部私钥。
- 请求、事件和日志序列化不泄露 token、私钥或代理密码。

### 事件与进程

- 前缀事件和普通输出混合时正确解析。
- 进度单调且跳过阶段不倒退。
- 无最终成功事件时，即使退出码为零也不报告成功。
- 取消能够终止进程树并进入 `cancelled`。
- 高频日志保持有界，错误事件不被普通日志淹没。

### Qt 面板

- 无参数脚本入口打开 GUI，程序调用 `main([])` 仍走无界面路径。
- 明暗主题、强调色、DPI 和自定义标题栏正常。
- 所有远端写选项默认不勾选。
- 本地调试模式禁用远端写选项。
- 高版本模式只解锁选项，不自动上传。
- 日志、进度、失败摘要和输出路径都在面板显示。
- 面板关闭或页面销毁时不访问已删除 Qt 对象。

### 发布回归

- 现有 `--build-only` 和正式发布测试继续通过。
- 构建一个低版本本地调试安装包，验证其版本与统一事实源一致，并包含现有生产公钥。
- 构建一个高版本发布候选，验证安装包、清单、签名、标签和文件名版本一致。
- 用已发布公钥验证生成的远端清单签名。

## 验收标准

1. 项目产品版本只需修改 `shared/version.py` 一处。
2. 面板“应用版本”后，GUI、WebUI、CLI、Python 包、安装器和更新清单读取同一值。
3. 低于远端版本的构建不会生成或上传签名资产，但安装包能够验证未来生产热更新。
4. 高于远端版本时显示发布选项，所有远端写操作默认关闭。
5. 构建、签名和上传不会阻塞 Qt UI 线程。
6. 构建日志实时显示在面板中，并有脱敏持久化副本。
7. 面板完整复用项目主题、自定义标题栏和原生窗口行为。
8. 私钥与 GitHub token 不进入仓库、安装包、事件流或日志。
9. 现有 CLI、CI 和发布测试入口保持兼容。
10. 相关焦点测试、架构守卫和发布打包回归全部通过。
