# 可选系统安装包签名校验设计

## 背景

桌面更新链路已经使用 Ed25519 对 `latest.json` 签名，并在安装前校验清单版本、安装包大小和 SHA-256。Windows Authenticode 是额外的系统发布者身份校验；当前个人发布阶段尚未购买受信任代码签名证书，因此空的发布者和证书指纹白名单会让已经通过自有更新校验的安装包仍被拒绝。

本次只暂缓系统安装包签名校验，不降低更新清单和文件完整性校验。保留 Authenticode 实现及配置接口，未来取得证书后可直接恢复。

## 方案

在 `app/config/update_trust.py` 增加显式布尔配置 `UPDATE_REQUIRE_OS_SIGNATURE`，当前设为 `False`。空白名单本身不代表关闭校验，避免配置遗漏或误删指纹时静默降低安全级别。

`PackageVerifier` 增加默认值为 `True` 的 `require_os_signature` 参数。其 `verify()` 始终先执行清单中大小与 SHA-256 对应的文件校验；只有文件完整性通过且开关启用时，才继续调用 Windows Authenticode 或 macOS 系统签名校验。该类的安全默认值保持不变，只有桌面应用和独立更新助手显式传入项目配置。

桌面 GUI 下载完成后会执行第一次校验，独立更新助手在父进程退出后、启动安装器前执行第二次校验。两个入口必须读取同一配置，防止 GUI 与 helper 的安全策略漂移。

## 信任边界

无论系统签名开关是否启用，下列检查都必须保持强制：

- `latest.json.sig` 必须由内置 Ed25519 公钥验证通过。
- 清单版本必须等于用户选择的目标版本。
- 当前客户端必须满足 `minClientVersion`。
- 下载地址必须符合受信主机和重定向策略。
- 安装包实际大小和 SHA-256 必须与已签名清单一致。
- 独立 helper 必须重新验证清单、目标版本和安装包完整性，不能信任 GUI 传入的未签名参数。

关闭 `UPDATE_REQUIRE_OS_SIGNATURE` 仅表示 Windows 不再要求 Authenticode 状态、发布者名称和证书指纹匹配。它不能绕过上述任一检查。

## 后续启用

取得正式代码签名证书后：

1. 将公开发布者名称写入 `UPDATE_TRUSTED_PUBLISHERS`。
2. 将公开证书 SHA-256 或 SHA-1 指纹写入 `UPDATE_TRUSTED_THUMBPRINTS`。
3. 将 `UPDATE_REQUIRE_OS_SIGNATURE` 改为 `True`。
4. 使用现有签名构建入口生成安装包，并运行更新链路测试和真实安装演练。

私钥、PFX/P12、令牌 PIN 和云签名凭据永远不得写入该配置或 Git。这里只保存客户端可公开分发的信任锚。

## 界面与日志

更新界面不再宣称当前执行了“系统签名校验”，统一描述为“更新清单签名、大小和 SHA-256 校验”。内部注释会说明 Authenticode 扩展点仍保留。现有下载、取消、重试、安装和重启交互不变。

## 测试

- 新增回归测试，证明关闭系统签名时不会调用 Authenticode 命令，但仍拒绝 SHA-256 不匹配的安装包。
- 保留并运行现有开启系统签名时的白名单、指纹、发布者和无效签名测试。
- 验证 GUI 和独立更新助手都显式传递同一个配置开关。
- 运行更新检查、更新器、打包、UI 文案、乱码保护和静态检查测试。

## 非目标

- 不移除 Authenticode 或 macOS 系统签名实现。
- 不允许未签名的 `latest.json` 或仅依赖 GitHub Release 数据更新。
- 不修改版本选择、断点续传、安装器参数或重启协议。
- 不把系统签名开关暴露为普通用户设置，避免运行时随意降低安全级别。
