# 同版本修订与可选热更新协议设计

## 状态

- 日期：2026-07-21
- 状态：已确认，待实施
- 决策：修订在 GitHub 上独立发布，在客户端中按产品版本逻辑聚合

## 背景

现有更新链路只比较三段式产品版本号。`3.6.21` 已发布后，即使重新上传不同安装包，已经安装
`3.6.21` 的客户端仍会把远端版本判定为“没有更新”。同时，覆盖原标签或原 Release 资产会破坏
源码提交、签名清单和安装包之间的可追溯关系。

项目需要长期保留以下能力：

1. 产品对外版本号不变时，可以连续发布多个修订。
2. 用户可以查看并选择目标修订，默认选中最新修订。
3. 更新始终由用户确认，不静默安装，也不强制安装。
4. 每个修订都能追溯到唯一源码提交，并单独完成签名、校验和安装回执。
5. 未来可以启用 GitHub Immutable Releases，而不需要再次改造发布模型。

本设计取代旧发布面板设计中“修改或覆盖同一个 GitHub Release”的同版本修复语义。旧设计的
单一产品版本事实源、构建子进程、密钥隔离和远端回读校验等约束继续有效。

## 目标与非目标

### 目标

- 建立 `(产品版本, 修订号)` 组成的发布身份。
- 同一产品版本的多个修订在 GUI 和 WebUI 中聚合展示。
- 每个修订使用独立 Git 标签、GitHub Release、安装包、清单和签名。
- 自动检查只提示严格高于当前安装身份的候选项。
- 用户可以显式重新安装当前修订，或在二次确认后回退到旧修订。
- 安装成功判断同时校验产品版本和修订号。
- 兼容没有修订字段的旧清单和旧安装。

### 非目标

- 不移动或覆盖已经发布的标签。
- 不删除、替换已经发布的修订资产。
- 不把修订号写回 `shared/version.py`，产品版本仍保持单一事实源。
- 不绕过 Ed25519 清单签名、资产 SHA-256 或现有安装前校验。
- 不把同版本修订变成强制更新机制。

## 发布身份

新增不可变值对象 `ReleaseIdentity`：

```text
ReleaseIdentity(version=SemVer("3.6.21"), revision=2)
```

身份排序规则为：

1. 先比较规范化 SemVer。
2. 产品版本相同时比较非负整数修订号。

修订号 `0` 表示该产品版本的初始发布。没有修订元数据的旧安装和旧清单都解释为修订 `0`。
修订号只允许递增，不复用已经发布的编号。

### 标签与显示

| 身份 | Git 标签和 Release tag | 用户显示 |
| --- | --- | --- |
| `3.6.21 / 0` | `v3.6.21` | `3.6.21 初始版` |
| `3.6.21 / 1` | `v3.6.21-r1` | `3.6.21 修订 1` |
| `3.6.21 / 2` | `v3.6.21-r2` | `3.6.21 修订 2` |

产品主界面的紧凑版本文本可以显示为 `v3.6.21-r2`；更新详情使用完整中文文本。修订标签必须指向
本次构建的 `sourceCommit`，不得沿用产品初始标签所指向的旧提交。

## GitHub 发布拓扑

每个修订创建独立 GitHub Release，但客户端把相同产品版本的 Release 聚合为一组：

```text
GitHub Releases                      更新面板

v3.6.21-r2  latest.json  installer   3.6.21
v3.6.21-r1  latest.json  installer     - 修订 2（最新）
v3.6.21     latest.json  installer     - 修订 1
                                          - 初始版
```

每个 Release 都只包含该修订自己的资产：

- 安装包
- `latest.json`
- `latest.json.sig`
- 发布流程要求的公开信任材料

资产发布后只读。重试上传时，远端同名资产与本地摘要一致则跳过；不一致则失败并要求创建新修订，
不得用 `--clobber` 覆盖历史资产。

独立 Release 让标签、源码提交和资产保持一一对应，也允许仓库将来启用 GitHub Release 不可变
保护。客户端的“同一个版本”体验由聚合层提供，不依赖远端使用一个可变资产容器。

## 签名清单

现有清单增加可选字段：

```json
{
  "schema": 1,
  "version": "3.6.21",
  "releaseRevision": 2,
  "tag": "v3.6.21-r2",
  "sourceCommit": "<40-character commit id>",
  "assets": []
}
```

`releaseRevision` 缺失时按 `0` 处理。继续使用 schema 1，是因为新增字段对旧清单解析器是向后兼容的；
签名覆盖包含该字段的完整规范化清单，攻击者不能修改修订号而不使签名失效。

解析器必须验证：

- 修订号是非负整数，布尔值不视为整数。
- 修订 `0` 的标准标签是 `v<version>`。
- 正修订的标准标签是 `v<version>-r<revision>`。
- 清单标签、GitHub Release tag、候选身份和安装请求完全一致。
- 同一身份出现两个不同的已签名清单时，判为远端冲突，不自动选择其中任意一个。

签名无效或资产摘要不匹配的 Release 不进入候选列表。检查结果应保留完整性告警，不能把“发现
无效的更高修订”静默显示成“已经是最新版”。

## 构建与发布流程

### 新产品版本

目标产品版本高于远端最高产品版本时，默认发布修订 `0`：

1. 应用并提交产品版本变更。
2. 创建 `v<version>` 标签。
3. 构建带修订 `0` 身份的便携版和安装包。
4. 生成、签署并回读校验该修订清单。
5. 创建独立 Release 并上传资产。

### 同版本修订

目标产品版本等于远端最高产品版本时，“同版本修复”模式执行：

1. 查询该产品版本所有合法修订，计算 `max(revision) + 1`。
2. 面板显示将要创建的修订号和标签，例如 `修订 3 / v3.6.21-r3`。
3. 要求工作树提交完成，修订标签必须指向当前 HEAD。
4. 构建产物写入新的修订目录，不能复用或覆盖旧修订目录。
5. 创建独立标签和 Release，上传该修订自己的签名资产。
6. 回读 Release、清单、签名和安装包摘要后才报告发布成功。

自动计算修订号只是建议值。无头构建必须显式接收期望修订号，并在远端预检后拒绝重复编号或
跳号，避免两个发布进程竞争生成同一修订。远端状态未知时禁止正式修订发布，只允许本地构建。

### 产物中的安装身份

构建阶段在便携目录和安装目录中写入只读的 `release_identity.json`：

```json
{
  "version": "3.6.21",
  "revision": 2,
  "tag": "v3.6.21-r2",
  "sourceCommit": "<40-character commit id>"
}
```

该文件由构建流程产生，不写入源码工作树。开发态找不到该文件时使用源码产品版本和修订 `0`。
Windows 文件版本和安装器内部版本可以投影为四段数值 `3.6.21.2`，但应用对外产品版本仍是
`3.6.21`。

## 更新发现与选择

更新服务继续枚举最近的 GitHub Releases，逐个下载并验证清单，然后执行以下步骤：

1. 把已验证候选项转换为 `ReleaseIdentity`。
2. 按产品版本分组，每组按修订号降序排列。
3. 整体按产品版本降序排列。
4. 默认选择最高产品版本中的最高修订。

自动检查只有在远端身份严格高于当前安装身份时才提示更新。没有任何修订可以绕过用户确认。

旧清单中的 `mandatory` 字段继续参与签名并兼容读取，但不再授予强制安装权限。发布工具默认且仅生成
`mandatory=false`；客户端即使读取到历史 `mandatory=true`，也只能把它显示为“重要更新”，不能
自动下载、自动安装、隐藏关闭入口或绕过用户的“稍后提醒”选择。

### 交互状态

| 目标与当前的关系 | 主文案 | 主操作 |
| --- | --- | --- |
| 产品版本更高 | 发现新版本 | 更新到所选版本 |
| 产品版本相同、修订更高 | 发现同版本修复 | 安装修订 |
| 身份完全相同 | 当前修订已安装 | 重新安装 |
| 目标身份更低 | 将回退到旧版本或修订 | 回退并二次确认 |

GUI 和 WebUI 使用相同的更新状态模型：

- 版本选择器先展示产品版本，再展示该版本下的修订。
- 默认选中最新产品版本的最新修订。
- 当前安装身份有“当前”标记，远端最高身份有“最新”标记。
- 同版本修订明确显示修订号、发布日期和修复说明。
- 操作按钮提供“立即更新”“稍后提醒”“忽略此修订”。
- 忽略状态记录完整身份；忽略修订 2 后，修订 3 发布时仍会再次提示。
- 自动检查不默认展示旧修订；用户展开“其他修订”后才可选择重新安装或回退。
- 所有候选项都保留关闭和稍后处理能力，远端清单不能改变这项本地交互策略。

安装开始后关闭按钮和 Escape 行为继续遵守现有安装状态机，不允许用户把“正在交接给安装器”误当
成已经取消。界面不得把下载完成、辅助进程启动和安装成功合并成一个状态。

## 安装交接与启动健康检查

`PreparedUpdate`、辅助安装进程参数和 `PendingInstall` 都必须携带完整目标身份。辅助进程在启动
安装包前验证：

- 清单签名和资产摘要有效。
- 目标版本、修订、标签与准备阶段一致。
- 安装包来自已批准的暂存目录。

应用重启后读取安装目录中的 `release_identity.json`，只有运行身份与待安装身份完全相等时才记录
安装成功。仅产品版本相同但修订不同，不得清除 pending 状态或报告成功。

用户显式选择回退时，更新服务生成带 `allowDowngrade=true` 的一次性准备结果。该权限不能来自远端
清单，也不能用于自动检查；辅助进程只接受由当前本地会话产生并绑定到目标清单摘要的回退授权。

## 本地状态兼容

更新状态文件扩展为保存完整身份：

- `lastSeenVersion` + `lastSeenRevision`
- `skippedVersion` + `skippedRevision`
- `pendingInstall.version` + `pendingInstall.revision`

旧状态缺少修订号时按 `0` 迁移。读取兼容旧字段，下一次成功保存时写入新结构，不单独批量改写用户
文件。

## 首次迁移限制

已经发布的旧客户端只会比较产品版本，不认识修订号。因此，已经安装旧版 `3.6.21` 的客户端无法
自行发现 `3.6.21-r1`。这是客户端能力的启动依赖，不能通过重新上传服务器资产解决。

正式迁移采用以下规则：

1. 第一个支持修订协议的公共版本必须使用更高的产品版本发布，例如 `3.6.22` 修订 `0`。
2. 用户升级到该版本后，后续 `3.6.22-r1`、`3.6.22-r2` 才能被同版本检测发现。
3. 尚未对外分发的测试安装，可以通过手动安装协议版本进入新模型。

发布面板应在首次启用修订协议且远端基线不支持时给出明确提示，不能声称旧同版本客户端能够收到
修订更新。

## 安全边界

- 私钥继续只从仓库外路径读取，不进入构建目录、日志、Git diff 或 Git index。
- 每个修订独立签署清单；GitHub token 泄露不能伪造合法修订。
- 发布端拒绝覆盖远端不一致资产，拒绝移动已发布标签。
- 客户端只安装签名有效且摘要匹配的候选项。
- 版本或修订回退只能由本地用户显式发起，并显示二次确认。
- OS 代码签名校验接口继续保留；未配置正式证书时不伪造发布者身份。

## 错误与恢复

- 某个候选清单无效：排除该候选并显示完整性告警，其他合法候选仍可查看。
- 远端出现重复身份：阻止该身份安装，提示维护者处理冲突。
- 修订号竞争：发布预检失败，重新查询后生成下一个修订，不覆盖已有修订。
- 上传中断：保留本地已验证产物；重试只补传缺失且名称未冲突的资产。
- 安装未完成：保留 pending 状态和暂存诊断，不把同产品版本误判为成功。
- 当前身份文件损坏：更新检查显示“安装身份未知”，允许修复安装，但不自动降级。

## 测试策略

### 身份与清单

- 修订缺失解释为 `0`。
- 拒绝负数、浮点数、布尔值和标签不匹配的修订。
- `(version, revision)` 排序、相等和序列化稳定。
- 清单签名覆盖修订号，篡改修订后验证失败。

### 发现与策略

- 同产品版本多个 Release 被聚合并按修订降序展示。
- 默认选择全局最新产品版本的最新修订。
- 当前修订不会触发自动更新，更高修订会触发。
- 忽略某一修订不屏蔽后续修订。
- 无效高修订不会被安装，也不会被静默吞掉。

### 构建与发布

- 新产品版本生成修订 `0` 标签。
- 同版本修复生成下一个修订标签和独立输出目录。
- 重复或跳跃修订在远端预检失败。
- 发布工具不移动旧标签、不覆盖旧资产。
- 回读验证确认标签、提交、清单、签名和安装包摘要一致。

### 安装与恢复

- pending 身份必须同时匹配产品版本和修订号。
- 同版本不同修订不会误报安装成功。
- 用户显式回退可安装旧修订，自动检查不能触发回退。
- 旧状态和缺少身份文件的安装按修订 `0` 兼容。

### 双端交互

- GUI 与 WebUI 显示相同的当前身份、候选分组和默认选项。
- 同版本修复、新版本、重新安装和回退使用不同文案与按钮状态。
- “稍后提醒”和“忽略此修订”不会启动下载。
- 安装交接阶段不能通过关闭按钮或 Escape 伪装成取消成功。

## 验收标准

1. 发布 `v3.6.22-r1` 后，已安装协议版 `3.6.22` 的客户端能发现但不会强制安装。
2. 更新面板默认选中最新修订，并允许查看该产品版本的其他修订。
3. 用户选择修订后，下载、校验、辅助安装和启动健康检查使用同一完整身份。
4. 每个修订都有独立标签、Release、签名清单和安装包，旧资产不被覆盖。
5. GUI 与 WebUI 的修订展示、忽略、重新安装和回退语义一致。
6. 旧清单、旧状态和旧安装按修订 `0` 兼容。
7. 私钥、PFX、P12、密码和 secret 文件不进入 Git diff、Git index 或构建日志。
