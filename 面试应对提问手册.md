## 1. 项目概览与架构设计

### 核心提问

**Q1: 用一分钟介绍你的项目。**

参考要点：
- 一句话定位：Windows 桌面多平台媒体采集工具，Python + PyQt6 + Playwright + FastAPI
- 核心能力：5 平台采集（抖音/B站/快手/小红书/missav）、统一下载调度、本地资产管理、GUI + WebUI 双模式
- 工程规模：v3.6.17、131 测试文件、413 passed、Docker 化、PyInstaller 打包
- 技术亮点：自研 EventBus、4 级下载回退、Ed25519 签名更新、反检测体系、平台签名算法

**Q2: 为什么选择 PyQt6 而不是 Electron?**

参考要点：
- 原生性能：PyQt6 是 C++ 绑定，渲染和事件循环走 Qt 原生，不启动 Chromium 实例
- 内存占用：Electron 基础内存 150MB+，PyQt6 约 40MB
- 与 Python 生态无缝：爬虫、下载、加密都是 Python 库，无需 IPC 桥接
- 打包体积：PyInstaller 单目录约 80MB（含 Playwright 浏览器），Electron 至少 120MB
- 代价：Qt 的 QSS 样式不如 CSS 灵活，跨平台一致性不如 Electron

**Q3: 你的项目分层架构是怎样的?**

参考要点：
```
entry/         薄入口层（gui/cli/web/interactive/dispatcher）
app/
  controllers/ 控制器 mixins（application/download/crawl/media/debug）
  services/    领域服务（frontend_state/cache/secure_updater/...）
  core/
    spiders/   平台 spider + base
    downloaders/ 下载策略链
    guardrails/  防护栏（budget/rate_limiter/pii）
    anti_detection/ 反检测
    plugins/   插件注册表
  ui/          PyQt6（pages/components/viewmodels/layout）
  web/         FastAPI（rest_router/ws_router/session_runtime）
  utils/       工具
cli/           命令行 + SDK + pipe
shared/        跨入口共享运行时
```

**Q4: 为什么用 mixins 而不是继承做控制器?**

参考要点：
- `ApplicationController` 需要组合多个能力（下载、爬虫、媒体、调试、会话），但 Python 单继承
- Mixin 让每个能力独立测试：`test_download_controller_mixin.py`、`test_crawl_controller_mixin.py`
- 避免 God Class：`application_controller.py` 只做编排，不实现细节
- 代价：mixin 之间有隐式依赖（共享 `self._event_bus`），需要文档约束

**Q5: BaseSpider 基类如何统一平台 spider 的线程协议与防护栏?**

参考要点（`d:\desktop\project\UniversalCrawlerProplus\app\spiders\base.py`）：
- `BaseSpider(threading.Thread)`（行 18-828）：线程基类，`daemon=True`、`name=self.__class__.__name__`，统一承接 UI 回调、中断语义和防护栏；文档串明确"平台 Spider 的线程基类"
- 5 个 `CallbackSignal` 信号（行 30-34）：`sig_log`/`sig_item_found`/`sig_items_found`/`sig_finished`/`sig_select_tasks`，用观察者模式解耦 spider 线程与 UI/CLI/Web 宿主
- 模板方法：`run()`（行 569-580）调用 `_run_impl()`（行 587-589，默认 `raise NotImplementedError`），任何退出路径都 `_emit_finished()`；各平台子类（Bilibili/MissAV）直接覆写 `run()` 编排自己的流水线
- 防护栏内建：`self.budget = self._build_crawl_budget(config)`（行 36）、`self.rate_limiter = self._build_rate_limiter(config)`（行 37），所有平台共享 CrawlBudget + RateLimiter
- `guard_request(source=None, *, cancel_check=None)`（行 719-737）同时消费 budget 与 rate_limiter，超限抛 `BudgetExhausted` / `RateLimitCancelled`
- 可中断协议：`interruptible_sleep(seconds, step=0.5)`（行 184-193）长等待切片轮询 `is_running`；`interruptible_page_wait`/`wait_for_selector`/`wait_for_load_state`/`playwright_goto`/`playwright_reload`（行 195-355）把 Playwright 阻塞调用切成小块，让 `stop()` 在步长间隙生效
- `ask_user_selection(items)`（行 780-796）/`resume_from_ui(selected_indices)`（行 798-801）：暂停 spider 线程发 `sig_select_tasks`，阻塞在 `_resume_event` 上等宿主回填
- `revive_for_partial_selection(collected_count, label, *, requires_browser=False)`（行 810-828）：停止后若已有候选，短暂恢复 `is_running=True` 完成选择并生成任务
- 类常量 `ALLOW_SYSTEM_PROXY_FALLBACK = False`（行 21），MissAV 覆写为 `True`（`missav\spider.py:15`）
- 不用 `abc.ABC`/`@abstractmethod`，而是模板方法 + `NotImplementedError` 兜底

技术细节：观察者模式（CallbackSignal）+ 模板方法 + 防护栏内置。`stop()`（行 121-129）设置 `is_running=False`、`_interrupt_requested=True`、唤醒选择等待、关闭跟踪的 Playwright 浏览器。

## 2. Python 语言基础与进阶

### 核心提问

**Q1: 解释 GIL 对你项目的影响。**

参考要点：
- GIL 让 Python 同一时刻只有一个线程执行字节码
- 项目中的影响：下载器是 IO 密集型，多线程仍能并行等待网络，但 CPU 密集的加密（SM3、Ed25519）不会因多线程加速
- 应对：`RateLimiter` 用 `threading.RLock` 保护 token bucket；`EventBus` 的异步队列用独立线程消费
- 如果要 CPU 并行（如批量加密），应该用 `multiprocessing`，但项目没这个需求

**Q2: 你的项目怎么管理异步任务？asyncio 还是线程？**

参考要点：
- 双轨制：
  - Web 层（FastAPI）：`asyncio`，`asyncio.Semaphore` 控制 API 并发
  - 下载层：`threading.Thread` + `threading.BoundedSemaphore`，因为下载器有同步阻塞调用（requests、subprocess）
- 桥接：`FrontendStateService` 用 `subscribe_async` + `_queue_on_ui` 把异步事件投递到 Qt 事件循环
- 不混用：不在异步函数里直接调 `time.sleep`，用 `asyncio.sleep`

**Q3: 解释你项目中 `dataclass` 的使用。**

参考要点：
- `@dataclass(frozen=True, slots=True)`：`AntiDetectionContext`、`RateLimitProfile`、`DownloadRequest`
- `frozen=True`：不可变，防止运行时被修改，可做字典 key
- `slots=True`：固定属性集合，省内存，防止动态属性注入
- `field(default_factory=dict, init=False, repr=False)`：`CrawlBudget` 的内部锁和计数器不暴露给构造函数

**Q4: `contextvars` 在你项目里用在哪？**

参考要点：
- `EventBus._publish_depth = contextvars.ContextVar("publish_depth", default=0)`
- 用途：递归发布保护，防止 handler 内再次 publish 导致栈溢出
- 为什么不用 `threading.local`：`contextvars` 对 asyncio 协程也隔离，`threading.local` 只隔离线程

**Q5: 解释 `__init__` vs `__new__`，你项目哪里用到？**

参考要点：
- `__new__` 创建实例，`__init__` 初始化实例
- 项目没有自定义 `__new__`，但有 `__post_init__`（dataclass 的初始化后钩子）
- 单例模式（如 `ConfigManager`）如果要做，会用 `__new__`

**Q7: 平台特定 RateLimitProfile 怎么定义？`for_platform` 工厂如何按平台选择速率？`acquire` 令牌桶补充算法的细节是什么？**

参考要点（[app/core/guardrails/rate_limiter.py](file:///d:/desktop/project/UniversalCrawlerProplus/app/core/guardrails/rate_limiter.py)）：
- **RateLimitProfile 数据结构**（[L20-L23](file:///d:/desktop/project/UniversalCrawlerProplus/app/core/guardrails/rate_limiter.py#L20-L23)）：`@dataclass(frozen=True)`，仅两个字段 `tokens_per_second: float` 与 `burst: float = 1.0`，是不可变值对象；事实：该 dataclass 在文件内只定义、未被 `RateLimiter.__init__` 直接使用（后者直接接收两个标量参数），更像是字段约定的文档化载体，且未在 `__init__.py.__all__` 导出
- **平台特定配置 RESILIENCE_PROFILES**（[L11-L17](file:///d:/desktop/project/UniversalCrawlerProplus/app/core/guardrails/rate_limiter.py#L11-L17)）：`dict[str, float]`，五个内置平台速率：`douyin=0.5`、`bilibili=1.0`、`kuaishou=0.5`、`missav=2.0`、`xiaohongshu=0.5`（单位"个/秒"）。事实：字典值是单个 float（令牌速率），并非 `RateLimitProfile` 实例——命名与类型存在不一致
- **`for_platform(cls, platform)` 工厂**（[L47-L50](file:///d:/desktop/project/UniversalCrawlerProplus/app/core/guardrails/rate_limiter.py#L47-L50)）：对平台名做 `str(platform or "").lower()` 归一化；未命中字典的平台默认 `rate=1.0`；`burst` 不独立配置，固定取 `max(1.0, rate)`——例如 missav：rate=2.0/burst=2.0；douyin：rate=0.5/burst=1.0（因 `max(1.0, 0.5)` 兜底）
- **构造与不变量**（[L26-L45](file:///d:/desktop/project/UniversalCrawlerProplus/app/core/guardrails/rate_limiter.py#L26-L45)）：`tokens_per_second` 必须严格大于 0 否则抛 `ValueError`；`burst = max(1.0, float(burst))` 兜底保证桶容量 ≥ 1.0；桶初始即满 `self._tokens = self.burst`；时间源与睡眠函数可注入（`monotonic`/`sleep` 参数，用于测试 `ManualClock` 桩）；线程安全用 `threading.RLock()`（可重入锁）
- **acquire 令牌桶算法**（[L52-L76](file:///d:/desktop/project/UniversalCrawlerProplus/app/core/guardrails/rate_limiter.py#L52-L76)）：
  - `needed = max(0.01, float(tokens))`：每次请求至少 0.01 个令牌，避免 0 令牌死循环
  - 主循环 `while True`：先持锁调 `_refill_locked`；成功获取返回 `True`；否则释放锁后再做取消检查与睡眠（锁的颗粒度：补充与扣减持锁，睡眠与取消检查在锁外，避免长时间持锁阻塞其他线程）
  - **补充公式**（`_refill_locked`）：`tokens = min(burst, tokens + elapsed * tokens_per_second)`——基于单调时钟流逝时间线性补充，被 `burst` 封顶；补充后更新 `_updated_at = now`
  - 不足时返回 `(False, (needed - tokens) / tokens_per_second)`，即预计还需等待的秒数
  - **取消机制**：`cancel_check()` 在锁外调用，返回真值则 `acquire` 返回 `False`（不抛异常）
  - **睡眠切片**：`sleep(min(max(wait_seconds, 0.01), 0.25))`——单次睡眠裁剪到 [0.01, 0.25] 秒区间，确保取消信号能在 250ms 内被响应
- **测试覆盖**（[tests/unit/app/core/guardrails/test_runtime_policies.py#L109-L179](file:///d:/desktop/project/UniversalCrawlerProplus/tests/unit/app/core/guardrails/test_runtime_policies.py#L109-L179)）：`ManualClock`（[L18-L28](file:///d:/desktop/project/UniversalCrawlerProplus/tests/unit/app/core/guardrails/test_runtime_policies.py#L18-L28)）注入 `monotonic`/`sleep`；`test_rate_limiter_platform_specific_profiles` 验证 douyin/missav 取 RESILIENCE_PROFILES、未知平台默认 1.0；`test_rate_limiter_token_bucket_refill` 验证 2.0 tokens/s 等待约 0.5 秒后可再 acquire；`test_rate_limiter_concurrent_access_thread_safe` 20 线程并发 acquire burst=20 全部成功无残留线程

技术细节：令牌桶（线性补充+burst 封顶）+ 平台速率字典（for_platform 工厂）+ 可取消等待（cancel_check 锁外+睡眠切片 [0.01,0.25]）+ 可注入时钟（测试桩）+ RLock 线程安全。

## 4. 平台签名算法与逆向

### 核心提问

**Q1: 抖音 aBogus 签名的原理是什么？**

参考要点（`app/core/lib/douyin/encrypt/aBogus.py`）：
- 基于 `gmssl.sm3`（国密 SM3 哈希）
- 输入：UA + 请求参数 + 时间戳 + 浏览器指纹
- `__browser` 字段：`1536|742|1536|864|...|Win32`（屏幕分辨率 + 平台）
- `__reg`：8 个 32 位常量（SM3 初始哈希值）
- 输出：`cus` 结尾的字符串
- 为什么用 SM3：抖音国内版用国密算法，不是 MD5/SHA

**Q2: Bilibili WBI 签名怎么实现？**

参考要点（`app/utils/bilibili_wbi.py`）：
- 从 `nav` 接口获取 `img_url` 和 `sub_url`
- 提取 `img_key` 和 `sub_key`（各 32 字符）
- 用 `MIXIN_KEY_ENC_TAB` 打乱位置生成 `mixin_key`（64 字符）
- 参数排序 + 过滤 `!'()*` 字符
- `w_rid = MD5(query_string + mixin_key)`
- `MIXIN_KEY_ENC_TAB` 是官方前端混淆表，顺序不可改

**Q3: 小红书签名为什么复杂？**

参考要点（`app/spiders/xiaohongshu/sign.py`）：
- 自定义 Base64 字母表（`CUSTOM_BASE64_ALPHABET`、`X3_BASE64_ALPHABET`）
- MD5 + XOR + 时间戳组合
- `ENV_TABLE` 和 `ENV_CHECKS_DEFAULT`：环境检测字节数组
- `SIGNATURE_DATA_TEMPLATE`：`x0`(版本) / `x1`(平台) / `x2`(系统) / `x3` / `x4`
- `B1_SECRET_KEY = "xhswebmplfbt"`
- 生成 `XYS_` 前缀的签名串

**Q5: Bilibili WBI 的 MIXIN_KEY_ENC_TAB 混淆表与 mixin_key 生成具体如何工作?**

- 编码细节：MD5 输入为"排序后查询字符串 + mixin_key"，输出 32 位 hex；`usedforsecurity=False` 标注非安全用途以适配 FIPS 合规环境

## 5. 下载系统与容灾设计

### 核心提问

**Q1: 你的 4 级下载回退链是什么？**

参考要点（`strategy.py`）：
```
M3U8DownloadStrategy → ChunkedDownloadStrategy → FFmpegDownloadStrategy → HttpDownloadStrategy
```
- M3U8：流媒体优先，用 `N_m3u8DL-RE` 下载
- Chunked：分块下载，支持断点续传
- FFmpeg：通用媒体下载，用 ffmpeg

**Q11: HLS 本地代理解决了什么问题？整体流程是什么？**

参考要点（`app/core/downloaders/hls_proxy.py`）：
- 场景：N_m3u8DL-RE 下载 HLS 流时，部分平台（抖音/B站）对 `.ts` 分片请求强制校验 Referer/Origin/Cookie，而 N_m3u8DL-RE 作为外部进程无法灵活注入这些头
- 方案：在 127.0.0.1 起一个 `ThreadingHTTPServer`（`_LocalHlsProxy`），把原始 m3u8 URL 改写成 `http://127.0.0.1:<port>/hls?u=<base64(upstream_url)>` 交给 N_m3u8DL-RE
- 完整流程（`hls_proxy.py`）：
  1. `start()` 绑定 `("127.0.0.1", 0)` 随机端口 + daemon 线程 `serve_forever`（L170-178）
  2. N_m3u8DL-RE 请求本地 URL → `_HlsProxyHandler.do_GET`（L123-139）→ `upstream_url_from_path` 解 base64 token 还原原始 URL（L191-199）
  3. `serve()` 用 downloader 的 `_hls_proxy_open_upstream` 带完整 headers（Referer/Cookie）回源（L217-224）
  4. 响应是 m3u8 playlist → `rewrite_hls_playlist_for_proxy` 重写所有分片 URL 为本地代理 URL（L89-102），递归处理子 playlist
  5. 响应是 `.ts`/`.m4s` 分片 → `_stream_response_body` 流式转发（L268-286）
- URL token 编解码：`base64.urlsafe_b64encode` + `urllib.parse.quote`（L188-189），解码失败抛 `ExternalToolError`（L199）
- playlist 重写：`#` 开头的 tag 行用正则 `URI=([\"'])(.*?)(?:\1)` 替换 `URI=` 属性里的地址（L105-112），非 tag 行直接 `urljoin` 后改写
- 安全：只监听 127.0.0.1 回环，不暴露网络；`DomainPolicyEngine` 注入校验回源域名（L153, L207）

**Q12: HLS 代理怎么跟踪下载进度？为什么进度映射到 10-95 区间？**

参考要点（`hls_proxy.py` L288-316）：
- 三个计数器（`threading.Lock` 保护，因 `ThreadingMixIn` 多线程处理请求）：
  - `_segment_total`：从 m3u8 playlist 解析分片数（`count_hls_media_entries`，L76-86），跳过 `#` 开头和 `.m3u8` 子 playlist
  - `_segment_completed`：每完成一个分片转发 `_record_resource_complete` +1（L303-306）
  - `_bytes_served`：累计转发的字节数（L299-301）
- `progress_snapshot()` 返回 `(progress, bytes_served)`（L308-316）：
  - `progress = 10 + int(min(segment_completed, segment_total) * 85 / segment_total)`
  - 映射到 10-95 区间，最后 `min(95, max(10, progress))` 双向夹紧
  - 为什么不是 0-100：代理只负责回源，前后还有 N_m3u8DL-RE 的初始化（解析 m3u8）和合并阶段（ts→mp4），预留 0-10 和 95-100 的缓冲，避免进度回退
- 边界：`segment_total <= 0`（playlist 无分片）时返回 `(0, bytes_served)`，避免除零
- playlist 记录：`_record_playlist` 用 `max` 取较大值（L293），防止变码率 playlist 切换时总片数跳变

**Q13: HLS 代理的生命周期和线程安全怎么管理？**

参考要点（`hls_proxy.py` L115-185）：
- 服务器类：`_ThreadingHlsProxyServer(ThreadingMixIn, HTTPServer)`（L115-117）
  - `daemon_threads = True`：处理请求的子线程是 daemon，防止卡死
  - `allow_reuse_address = True`：避免 TIME_WAIT 导致端口复用失败
- 生命周期：
  - `start()`：创建 server + daemon 线程 `ucp-hls-proxy`，返回 self 供链式调用（L169-178）
  - `stop()`：`server.shutdown()` 停止 `serve_forever` + `server.server_close()` 关闭 socket + `thread.join(timeout=2)`（L180-185）
  - daemon=True：进程退出时自动回收，`join(timeout=2)` 防止卡在 shutdown
- 端口绑定 `("127.0.0.1", 0)`：回环地址 + OS 分配随机端口，安全且无冲突（L170）
- 请求处理：`_HlsProxyHandler.do_GET` 捕获所有异常，记 `debug_logger.log_exception` 后返回 502（L131-139）
- `log_message` 重写为空（L141-142）：禁用 `BaseHTTPRequestHandler` 默认的 stderr 日志，避免污染管道输出和日志噪音
- 流式响应：`_stream_response_body` 逐 chunk `wfile.write` + `flush`，`BrokenPipeError`/`OSError` 捕获后只记日志不外抛（L278-282），防止 N_m3u8DL-RE 主动断开导致代理崩溃
- 响应头透传：`Content-Length`/`Content-Range`/`Accept-Ranges`/`ETag`/`Last-Modified`/`Cache-Control` 原样转发（L246-256），支持分片 Range 请求
- 上游响应关闭：`finally` 块确保 `response.close()`，异常只记日志（L260-266）

**Q14: 文件落盘遇到 Windows 瞬态锁冲突时如何统一重试?**

参考要点（`app/services/file_service.py`）:

## 6. EventBus 事件总线设计

### 核心提问

**Q1: 为什么自研 EventBus 而不用现成的？**

参考要点：
- 现成的（如 blinker、pydispatcher）不支持异步队列
- 需要 noisy topic latest-state-wins 合并，防止 UI 刷新队列堆积
- 需要递归保护、慢 handler 告警、条件变量 idle 等待
- Qt 信号槽跨线程开销大，EventBus 更轻量

**Q2: noisy topic 合并是什么意思？**

参考要点：
- `ASYNC_NOISY_TOPICS`：`videos.update`、`logs.append`、`task_progress` 等高频事件
- 按 entity ID 合并：同一视频的多次 `videos.update` 只保留最新
- `_AsyncTaskKey`：`(id(entity), topic, source, trace_id)` 作为合并 key
- `_async_pending_latest`：待投递的最新任务
- 效果：1000 条日志事件 → 1 条 UI 刷新

**Q3: 递归保护怎么实现？**

参考要点：
- `MAX_PUBLISH_DEPTH = 16`
- `contextvars.ContextVar("publish_depth")` 记录当前深度
- publish 时 `+1`，超过 16 抛 `RecursionError`
- 为什么用 contextvars：asyncio 协程也隔离

**Q4: `wait_for_async_idle` 怎么实现？**

参考要点：
- `threading.Condition(self._lock)` 条件变量
- `_async_inflight` 计数器跟踪在途任务
- 任务完成时 `notify_all`，等待者被唤醒
- 禁止用 `time.sleep` 或 `processEvents` 轮询
- 从异步线程调用时短路返回，防死锁

**Q5: 同步和异步订阅有什么区别？**

参考要点：
- `subscribe(topic, handler)`：同步，publish 时直接调用
- `subscribe_async(topic, handler)`：异步，投递到 `_async_queue`，独立线程消费
- 异步队列 `maxsize=1024`，溢出进 `priority_overflow` deque
- FIFO 保证（非 noisy topic）

**Q9: ConnectionRegistry 是什么？为什么用它代替直接 `signal.connect()`？**

参考要点（[app/ui/connection_registry.py](file:///d:/desktop/project/UniversalCrawlerProplus/app/ui/connection_registry.py)）：
- 定位：把 Qt 的 `signal.connect(slot)` 包一层登记簿，把 `(signal, QMetaObject.Connection, slot)` 三元组存进 `list`，便于窗口关闭时批量精确断开（[connection_registry.py#L18-L19](file:///d:/desktop/project/UniversalCrawlerProplus/app/ui/connection_registry.py#L18-L19)）
- RAII 双层设计：
  - 类级 `__enter__/__exit__`（[L68-L72](file:///d:/desktop/project/UniversalCrawlerProplus/app/ui/connection_registry.py#L68-L72)）：`with ConnectionRegistry() as reg: reg.connect(...)`，退出 `with` 块自动 `disconnect_all()`
  - `scoped()` 上下文管理器（[L34-L40](file:///d:/desktop/project/UniversalCrawlerProplus/app/ui/connection_registry.py#L34-L40)）：在已有 registry 上开子作用域，`start = len(self._connections)`，`finally` 只断开本作用域新增的连接（`_disconnect_from(start)`），不影响外部已注册的连接
- 批量断连采用**逆序**遍历（[L48](file:///d:/desktop/project/UniversalCrawlerProplus/app/ui/connection_registry.py#L48) `for ... in reversed(pending)`），符合栈式注册/释放惯例
- 实际使用：[app/ui/main_window.py#L216](file:///d:/desktop/project/UniversalCrawlerProplus/app/ui/main_window.py#L216) 创建实例，L238-278、L450-479 大量 `self._connections.connect(signal, slot)` 登记，[L2114](file:///d:/desktop/project/UniversalCrawlerProplus/app/ui/main_window.py#L2114) `closeEvent` 末尾 `disconnect_all()` 批量释放
- 为什么不直接 `signal.connect()`：Qt 的 `connect()` 返回 `QMetaObject.Connection` 句柄，若不集中管理，窗口关闭时只能依赖 `QObject` 析构链被动断开；半销毁状态下容易出现"信号触发到已 deleteLater 的对象"导致 0xC0000409 崩溃。集中登记 + 显式断开可保证关闭顺序可控

**Q10: ConnectionRegistry 的三级降级断连是怎么设计的？为什么要降级？**

参考要点（[app/ui/connection_registry.py#L51-L66](file:///d:/desktop/project/UniversalCrawlerProplus/app/ui/connection_registry.py#L51-L66)）：
- 三级断连策略（`_disconnect_one` 静态方法）：
  1. 第一级：`QObject.disconnect(connection)` —— 按 `QMetaObject.Connection` 句柄断开，最精确
  2. 第二级：失败则 `signal.disconnect(connection)` —— 按信号对象 + 句柄断开
  3. 第三级：再失败则 `signal.disconnect(slot)` —— 按信号对象 + slot 引用断开
- 每级失败都 `_LOGGER.warning(..., exc_info=True)` 记录，不外抛异常
- 为什么需要降级：Qt 在对象半销毁状态下（C++ 部分已释放但 Python wrapper 还在），`QObject.disconnect(connection)` 可能抛 `RuntimeError: Failed to disconnect signal`；按 slot 断开对 slot 已被 GC 的情况更鲁棒
- `_disconnect_from(start)`（[L45-L49](file:///d:/desktop/project/UniversalCrawlerProplus/app/ui/connection_registry.py#L45-L49)）：先切片 `pending = self._connections[start:]` + `del self._connections[start:]`（先从列表移除，防止重入），再逆序遍历 pending 调 `_disconnect_one`
- 设计意图：保证清理路径不抛异常，即使个别连接断开失败也能继续清理剩余连接，避免 `closeEvent` 卡死或崩溃

**Q12: `_ShortTaskRunnable` 为什么用 `weakref` 持有 panel？这个机制目前在生产中是否使用？**

参考要点（[app/ui/task_runtime.py#L225-L264](file:///d:/desktop/project/UniversalCrawlerProplus/app/ui/task_runtime.py#L225-L264)）：
- weakref 注入（[L240](file:///d:/desktop/project/UniversalCrawlerProplus/app/ui/task_runtime.py#L240)）：`self._panel_ref = weakref.ref(panel) if panel is not None else lambda: None`
- L241 注释明确说明设计意图：`# No parent: _ShortTaskRunnable is QRunnable (not QObject); panel via weakref, hard parent unsafe.`
- 为什么不能用强引用：
  - `_ShortTaskRunnable` 是 `QRunnable`（不是 `QObject`），不能给 Qt 对象当 child
  - 若直接持有 `panel` 强引用，即使 panel 的 C++ 部分已被 `deleteLater()` 销毁，Python 端的 `panel` 对象仍因 runnable 的强引用而保留，runnable 在线程池里继续 `emit` 信号到已销毁的 QObject，触发 "QMetaObject::invokeMethod: No such method" 或段错误（0xC0000409）
- 信号连接（[L243-L249](file:///d:/desktop/project/UniversalCrawlerProplus/app/ui/task_runtime.py#L243-L249)）：把 `_signals.progress` 连到 `panel.on_short_task_progress`、`_signals.finished` 连到 `panel.on_short_task_finished`，均用 `Qt.ConnectionType.QueuedConnection` 保证 slot 在 panel 所在线程（UI 线程）执行
- run() 防悬挂（[L252-L264](file:///d:/desktop/project/UniversalCrawlerProplus/app/ui/task_runtime.py#L252-L264)）：emit 前都先判空 `if self._panel_ref() is not None:`，panel 已销毁则跳过所有 emit
- 错误处理（[L258-L259](file:///d:/desktop/project/UniversalCrawlerProplus/app/ui/task_runtime.py#L258-L259)）：`except Exception: self._logger.exception(...)`，异常吞掉只记日志，保证线程池不被异常拖垮；`finally` 无论如何都 emit finished + `mark_done()`
- **生产现状**：该 weakref+panel-hook 机制目前是**休眠的预置协议**——全仓 grep `on_short_task_progress`/`on_short_task_finished`/`submit(...panel=...)` 均无生产调用方命中，测试 `tests/unit/app/ui/test_task_runtime.py` L113/L123/L128 也不传 `panel=`
- 当前生产实际走两条路径（见 Q13）：panel 自定义信号 + QueuedConnection，或 `_UiCallbackInvoker` 携带 callable
- 设计意义：任何想接 progress/finished hook 的 panel，只需实现两个 hook 方法并传 `panel=self` 即可获得 weakref 安全保障，是面向未来的扩展点

**Q14: 窗口几何恢复后为什么要"夹回屏幕"？多屏/DPI 切换场景怎么处理？**

参考要点：
- 痛点：`restoreGeometry` 只按上次保存的字节流恢复，不感知当前物理屏幕——如果用户上次在 2K 外接屏上保存几何，下次只接笔记本 1080P 屏，窗口会超出可视区域，标题栏甚至拖不出来

## 8. Playwright 浏览器自动化

### 核心提问

**Q1: 为什么用 Playwright 而不是 Selenium?**

参考要点：
- Playwright 原生支持异步，Selenium 是同步封装
- Playwright 的 `BrowserContext` 隔离更好，可注入 init script
- 自动等待机制：`wait_for_selector`、`wait_for_load_state`
- 多浏览器支持：Chromium/Firefox/WebKit
- API 更现代，Python 绑定完善

**Q2: stealth.js 怎么注入？**

参考要点（`stealth.py`）：
- `context.add_init_script(load_stealth_script())`
- 必须在 `context.new_page()` 之前注入
- `lru_cache(maxsize=1)` 缓存脚本内容
- `apply_to_context(context)`：统一入口，所有 spider 调用
- 时机：`context = browser.new_context()` → `apply_stealth_to_context(context)` → `page = context.new_page()`

**Q3: 快手瀑布流怎么模拟？**

参考要点（`kuaishou/spider.py`）：
- `window.scrollBy(0, 800)` 滚动
- `scroll_into_view_if_needed()` 元素滚入视图
- `page.keyboard.press("ArrowDown")` 键盘翻页
- `_capture_scroll_budget`：滚动预算控制
- `scroll_budget` 循环生产者：边滚动边采集

**Q4: 怎么处理浏览器停止？**

参考要点：
- `_playwright_lock` 保护 browser 引用
- `stop()` 时立即 `browser.close()`，不等页面加载完
- `_playwright_owner_thread_id`：只允许创建线程关闭
- 同步 API 可能卡在页面加载，`close()` 是唯一的中断方式

**Q5: 反检测上下文怎么管理？**

参考要点（`AntiDetectionContext`）：
- `source`：平台标识
- `user_agent`：UA

**Q8: WebSocketSessionBinder 为什么用 secrets.compare_digest?四套 cookie 各自的作用是什么?CSRF 怎么防护?**

参考要点（[app/web/ws_session_binding.py](file:///d:/desktop/project/UniversalCrawlerProplus/app/web/ws_session_binding.py)）：
- **类定位**：`WebSocketSessionBinder`（L19-57）负责从 WebSocket 请求恢复会话并完成鉴权；`WebSocketSessionBinding`（L12-17）dataclass with slots，承载 `session_id` 和 `context`
- **bind() 完整流程**（[ws_session_binding.py#L35-L57](file:///d:/desktop/project/UniversalCrawlerProplus/app/web/ws_session_binding.py#L35-L57)）：
  1. **访问令牌校验（可选）**（L36-40）：若配置了 `access_token`，从 `ucrawl_access_token` cookie 取值，`secrets.compare_digest(access_token or "", self._access_token)`，失败 `ws.close(code=1008, reason="invalid access token")`
  2. **恢复 session_id**（L41-42）：`ws.cookies.get("ucrawl_session") or ws.cookies.get("ucrawl_session_id") or self._default_session_id`，调 `session_registry.get_or_create(session_id)` 取/建 `WebSessionContext`
  3. **Origin 校验**（L43-50）：`expected_origin = f"{ws.url.scheme.replace('ws', 'http', 1)}://{ws.url.netloc}"`（ws→http 替换），`is_allowed_origin(origin, expected_origin=expected_origin)` 不通过 `ws.close(1008, "forbidden origin")`
  4. **会话令牌校验**（L44, L46, L51-55）：`token = ws.cookies.get("ucrawl_session_token")`，`token_valid = secrets.compare_digest(token or "", context.session_token)`；非本地请求（`origin` 存在或 `not is_local_host(client_host)`）且 token 无效则 `ws.close(1008, "invalid session token")`
- **secrets.compare_digest 使用位置**（全仓 5 处）：
  - WS 层：`ws_session_binding.py` L38（access token）、L46（session token）
  - HTTP 层（`http_session.py`）：L122（GET `?access_token=xxx` 重定向校验）、L145（`_has_valid_application_access` 校验 cookie 或 Bearer）、L157（`has_valid_session_token` 校验 CSRF token）
- **为什么时间安全比较**：普通 `==` 在第一个不匹配字节短路返回，攻击者可测响应时间逐字节猜测令牌（timing attack）。`secrets.compare_digest` 始终比较所有字节，执行时间与输入内容无关，阻断时序侧信道
- **四套 cookie**（常量定义 [app/web/server.py#L41-L45](file:///d:/desktop/project/UniversalCrawlerProplus/app/web/server.py#L41-L45)）：
  | Cookie | 常量 | httponly | samesite | 作用 |
  |---|---|---|---|---|
  | `ucrawl_session` | `SESSION_COOKIE_NAME` | True | strict | 会话 ID，标识 WebSessionContext |
  | `ucrawl_session_token` | `SESSION_TOKEN_COOKIE_NAME` | True | strict | 会话令牌，WS 与非本地 HTTP 校验 |
  | `ucrawl_csrf_token` | `CSRF_COOKIE_NAME` | **False** | strict | CSRF 令牌，JS 可读用于双重提交 |
  | `ucrawl_access_token` | `ACCESS_COOKIE_NAME` | True | strict | 应用级访问令牌 |
- **Token 生成**（`session_runtime.py` L105-106）：`session_token = secrets.token_urlsafe(24)`、`csrf_token = secrets.token_urlsafe(24)`，高熵随机
- **Cookie 签发**（`HttpSessionCoordinator.handle` http_session.py L43-99）：每次请求后补签——无 session cookie 则生成 `uuid4().hex` 签发（L75-82）；session_token 不匹配则签发（L83-90）；csrf_token 不匹配则签发（L91-98，`httponly=False` 因前端 JS 需读取做双重提交）
- **CSRF 防护（双重提交模式）**：`ucrawl_csrf_token` cookie 可被 JS 读取，前端通过 `X-Ucrawl-Session-Token` header 回传（`http_session.py` L156：`request.headers.get(self._session_token_header) or request.cookies.get(self._csrf_cookie_name)`）；写操作校验（L66-71）POST/PUT/DELETE 要求 Origin 合法且 token 有效，否则 403

技术细节：时间安全比较（`secrets.compare_digest` 防时序攻击）+ 四套 cookie 职责分离（session ID/session token/CSRF token/access token）+ CSRF 双重提交（cookie JS 可读 + header 回传）+ Origin 校验（WS 层 ws→http 替换推导 expected_origin）+ 本地豁免（非本地请求才强制 token 有效）。

**Q9: NoCacheStaticFiles 禁了哪些头?WebSocket bootstrap 为什么必须下沉 executor?WebUI 页面卸载如何停止 delta 推送?**

参考要点：
- **NoCacheStaticFiles**（`app/web/server.py`）：
  - 定义：`class NoCacheStaticFiles(StaticFiles)` 继承 FastAPI 的 `StaticFiles`，重写 `get_response`，在父类响应基础上附加三个禁缓存头
  - 三头：`Cache-Control: no-store, no-cache, must-revalidate, max-age=0`（HTTP/1.1 标准，`no-store` 禁存储 + `no-cache` 强制重新验证 + `must-revalidate` 禁用过期响应 + `max-age=0` 立即过期）、`Pragma: no-cache`（HTTP/1.0 兼容）、`Expires: 0`（HTTP/1.0 兼容立即过期）
  - 挂载：`create_app()` 在具体 API 路由之后挂载 `/static`；首页 `index.html` 通过 `_apply_no_cache_headers(FileResponse(...))` 使用同一份头部策略
  - 唯一实现：静态目录解析、首页响应与禁缓存策略均由 `server.py` 管理，架构守卫禁止旧 `static_router.py` 影子实现回流
  - 为什么禁缓存：① 开发态频繁改 JS/CSS，浏览器缓存导致调试用旧版本；② 打包态 PyInstaller 嵌入 exe，用户更新应用后旧浏览器缓存导致 JS/CSS 版本不一致（frontend_version 与 delta 协议不匹配引发渲染错误）；③ 前端逻辑复杂（generation/sequence/pageKey 隔离）对代码版本敏感
- **WebSocket bootstrap 下沉 executor**（[app/web/ws_bootstrap.py](file:///d:/desktop/project/UniversalCrawlerProplus/app/web/ws_bootstrap.py)）：
  - 核心辅助（L16-24）：`_run_controller_worker_call(func, *args)` = `await asyncio.get_running_loop().run_in_executor(None, func, *args)`（`None` 用默认 ThreadPoolExecutor）；`_send_json`（L22-24）连 JSON 序列化都下沉 executor

## 10. 数据库与持久化设计

### 核心提问

**Q1: 为什么用 SQLite 而不是 MySQL/PostgreSQL?**

参考要点：
- 桌面应用，不需要网络数据库
- SQLite 零配置，单文件，打包友好
- WAL 模式支持并发读
- 性能足够：单机场景写入 1000+ TPS
- 限制：不支持多进程写入，但项目不需要

**Q2: WAL 模式是什么？为什么用？**

参考要点：
- Write-Ahead Logging：写入先记日志，再合并到主库
- 优点：读写不互斥，并发性能好
- `synchronous=FULL`：每次提交都 fsync，防数据丢失
- `busy_timeout`：等锁超时，避免 `SQLITE_BUSY`
- 短连接 + `contextlib.closing()`：Windows 文件锁问题

**Q3: 恢复账本的数据模型是什么？**

参考要点（`download_recovery_store.py`）：
- `active` 行：任务创建时写入，含路径
- `pending_cleanup_directories`：失败目录，按路径去重
- 成功完成立即销账

## 11. 并发编程与线程安全

### 核心提问

**Q1: 你的项目用了哪些并发原语？**

参考要点：
- `threading.Thread`：spider 线程、下载 worker、EventBus 消费线程
- `threading.RLock`：EventBus、RateLimiter、CrawlBudget
- `threading.BoundedSemaphore`：下载并发控制
- `threading.Event`：停止信号、门闩
- `threading.Condition`：EventBus idle 等待
- `asyncio.Semaphore`：Web API 并发控制
- `queue.Queue`：EventBus 异步队列（maxsize=1024）

**Q2: RLock 和 Lock 的区别？**

参考要点：
- `Lock`：不可重入，同线程二次 acquire 会死锁
- `RLock`：可重入，同线程可多次 acquire（计数），需对应次数 release
- 项目用 `RLock`：`EventBus.publish` 可能调用 `subscribe` 的 handler，handler 可能再 publish
- 代价：性能略低于 `Lock`

**Q3: 条件变量怎么用？**

参考要点（EventBus idle 等待）：
```python
with self._lock:
    while self._async_inflight > 0:
        self._async_idle_condition.wait()
```
- `wait()` 释放锁并阻塞，被 `notify_all` 唤醒后重新获取锁
- 用 while 不用 if：防虚假唤醒
- 从异步线程调用时短路，防死锁

**Q4: 怎么避免死锁？**

参考要点：
- 锁层次：EventBus 用一把 `RLock`，不分多级
- 超时：`acquire(timeout=)`、`join(timeout=)`
- 不嵌套锁：handler 在锁外执行
- 条件变量复用主锁：`Condition(self._lock)`，不新建锁

**Q5: asyncio 和 threading 怎么协作？**

参考要点：
- `subscribe_async` + `_queue_on_ui`：异步事件投递到 Qt 事件循环
- `run_in_executor`：asyncio 调用同步函数
- 不混用：异步函数不调 `time.sleep`，同步函数不 `await`

**Q6: asyncio.Semaphore 和 threading.BoundedSemaphore 在你的项目里分别用在哪？为什么需要双层并发控制？**

参考要点：
- 两层信号量完全正交，分属不同并发模型，职责清晰：
  - Web 层（FastAPI）：`asyncio.Semaphore`，限制 API 操作并发数
  - 下载层：`threading.BoundedSemaphore`，限制下载 worker 槽位
- **asyncio.Semaphore**（[app/web/controller.py#L278](file:///d:/desktop/project/UniversalCrawlerProplus/app/web/controller.py#L278)、[L320-L321](file:///d:/desktop/project/UniversalCrawlerProplus/app/web/controller.py#L320-L321)）：
  - 类常量 `MAX_CONCURRENT_API_OPS = 16`（硬编码，不可配置）
  - 初始化 `self._api_operation_semaphore = asyncio.Semaphore(self.max_concurrent_api_ops)`
  - 唯一消费点 `_run_api_operation`（[controller.py#L1289-L1298](file:///d:/desktop/project/UniversalCrawlerProplus/app/web/controller.py#L1289-L1298)）：`async with self._api_operation_semaphore:` 包裹所有受保护 API 操作

**Q7: 下载层的 BoundedSemaphore 为什么需要运行时扩容？`_rebuild_slot_semaphore` 如何保证不超发？为什么用 Bounded 而不是普通 Semaphore？**

参考要点（[app/core/download_manager_core.py#L413-L432](file:///d:/desktop/project/UniversalCrawlerProplus/app/core/download_manager_core.py#L413-L432)）：
- 扩容场景：用户在运行时修改 `download.max_concurrent` 配置（如 3→5），热加载触发 `_rebuild_slot_semaphore(old_value, new_value)`
- `_rebuild_slot_semaphore` 算法三步（"吸干-统计-预占"）：
  1. 吸干旧信号量：循环 `slot_semaphore.acquire(blocking=False)` 直到失败，统计 `available_tokens`（空闲 token 数）
  2. 统计在用：`in_use_tokens = max(0, old_value - available_tokens)`，即已被 worker 占用的槽位
  3. 预占新信号量：创建 `BoundedSemaphore(new_value)`，循环 `next_semaphore.acquire(blocking=False)` 预占 `min(in_use_tokens, new_value)` 个 token，把正在运行的 worker 占用"迁移"到新信号量
  4. 替换引用 `self.slot_semaphore = next_semaphore`
- 仅支持扩容（new > old），不支持缩容——缩容会导致在用 worker 释放时新信号量超容
- `_acquire_dispatch_slot`（[L1019-L1038](file:///d:/desktop/project/UniversalCrawlerProplus/app/core/download_manager_core.py#L1019-L1038)）处理重建竞态：acquire 成功后检查 `semaphore is self.slot_semaphore`，若已被重建则 `release()` 旧 token 返回 False，防止旧信号量上的 token 算作有效占槽
- `_release_dispatch_slot`（[L1043-L1059](file:///d:/desktop/project/UniversalCrawlerProplus/app/core/download_manager_core.py#L1043-L1059)）双重释放防护：
  - worker 标记 `_slot_released`，已释放则跳过
  - `BoundedSemaphore.release()` 超容时抛 `ValueError`，捕获后跳过（重建期间旧信号量可能已超容）
- 为什么 Bounded 而非普通 Semaphore：`Bounded` 版本保证 `release` 次数不超过 `acquire` 次数，超容抛 `ValueError`；普通 `Semaphore` 不会报错，双重释放会导致信号量计数虚高，最终超发 worker。Bounded 把"释放 bug"变成可捕获的显式异常

技术细节：运行时扩容的核心难点是"不丢失在用 worker 的占用记录"——吸干旧信号量统计空闲数，反推在用数，再在新信号量上预占相同数量。"吸干-统计-预占"三步在 `_slot_semaphore_guard()` 锁保护下原子执行。

**Q8: RateLimiter、CrawlBudget、Semaphore 三者是什么关系？为什么设计成正交而不是嵌套？**

参考要点：
- 四重防护栏分层，各自独立、各自在调用链不同位置生效：

| 机制 | 控制维度 | 单位 | 作用层 | 原语 |
|------|---------|------|--------|------|
| `asyncio.Semaphore` | 并发数 | 个 | Web API 层 | asyncio |
| `threading.BoundedSemaphore` | 并发槽位 | 个 | 下载 worker 层 | threading |
| `RateLimiter` | 速率 | 请求/秒 | Spider 请求层（每平台一个） | threading.RLock 令牌桶 |
| `CrawlBudget` | 总量 | 累计请求 | Spider 任务生命周期 | threading.RLock 计数器 |

- RateLimiter（[app/core/guardrails/rate_limiter.py#L26-L45](file:///d:/desktop/project/UniversalCrawlerProplus/app/core/guardrails/rate_limiter.py#L26-L45)）：令牌桶，`tokens_per_second` + `burst`，`acquire(tokens, cancel_check)` 可取消等待；平台默认速率 `RESILIENCE_PROFILES`（[L11-L17](file:///d:/desktop/project/UniversalCrawlerProplus/app/core/guardrails/rate_limiter.py#L11-L17)）：douyin 0.5/s、bilibili 1.0/s、kuaishou 0.5/s、missav 2.0/s、xiaohongshu 0.5/s；`burst = max(1.0, rate)`
- CrawlBudget（[app/core/guardrails/crawl_budget.py#L18-L45](file:///d:/desktop/project/UniversalCrawlerProplus/app/core/guardrails/crawl_budget.py#L18-L45)）：`@dataclass`，默认 `max_requests_per_platform=1000`、`max_total=5000`；锁和计数器用 `field(init=False, repr=False)` 隐藏，保证 dataclass 不可变语义同时保留可变内部状态；`consume` 超限抛 `BudgetExhausted`
- RateLimiter 与 CrawlBudget 的唯一汇合点 `guard_request`（[app/spiders/base.py#L719-L737](file:///d:/desktop/project/UniversalCrawlerProplus/app/spiders/base.py#L719-L737)）：先 `budget.consume(platform)`（总量）再 `rate_limiter.acquire(cancel_check=...)`（速率），超限抛 `BudgetExhausted` / `RateLimitCancelled`
- `run()`（[base.py#L569-L578](file:///d:/desktop/project/UniversalCrawlerProplus/app/spiders/base.py#L569-L578)）捕获 `BudgetExhausted` 视为**优雅停止**不记错误
- 为什么正交不嵌套：
  - 语义不同：并发（同时进行）、速率（单位时间）、总量（累计），三者维度正交，无法相互替代
  - 作用层不同：API 入口/下载调度/请求前/任务生命周期，各自在调用链最合适的位置生效
  - 独立可测：RateLimiter、CrawlBudget 各有独立单测，互不依赖
  - 灵活组合：不同平台可配不同 RateLimiter，不同任务可配不同 CrawlBudget，Semaphore 全局共享
- 注意区分：`TransferRateLimiter`（[downloaders/base.py#L28](file:///d:/desktop/project/UniversalCrawlerProplus/app/core/downloaders/base.py#L28)）是下载**字节流速率限制**（`speed_limit_kb`），与请求级 RateLimiter 完全不同，作用在 worker 内部，与并发信号量无关

技术细节：Semaphore 控"同时进行"数（并发），RateLimiter 控"单位时间"请求数（速率，令牌桶允许短暂 burst），CrawlBudget 控"任务周期"累计量（总量上限）。三者互不调用，在 `guard_request` 汇合点分别检查 budget 与 rate_limiter，Semaphore 在更外层独立生效。

## 12. 安全工程与加密

### 核心提问

**Q1: Ed25519 签名怎么用在更新器？**

参考要点（`secure_updater.py`）：
- `Crypto.PublicKey.ECC` + `Crypto.Signature.eddsa`
- manifest（JSON）+ signature（.sig）双文件

- `DEFAULT_ALLOWED_HOSTS`：github.com 等白名单
- `DEFAULT_MAX_DOWNLOAD_BYTES`：512MB 上限

**Q2: SSRF 怎么防护？**

参考要点：
- `DomainPolicyEngine`：统一域名策略
- `ipaddress` 模块校验：禁止内网 IP（127.0.0.1、10.x、192.168.x）
- 主机白名单：只允许 github.com 等
- 重定向跟踪：`REDIRECT_STATUS_CODES` 集中管理

**Q3: Cookie 怎么安全处理？**

参考要点（`app/services/auth_service.py`）：
- `AuthService.extract_cookie_list/extract_cookie_dict/build_cookie_string` 是 Cookie 载入、规范化与请求头构建的唯一入口，兼容列表、Playwright storage-state 和字典三种输入形态。
- `_safe_to_string` 隔离异常对象，`required_cookie` 可在关键认证 Cookie 缺失时拒绝生成请求头，避免带着半认证状态继续请求。
- JSON 读写异常统一转换为 `CookieLoadError` / `CookieSaveError`，调用方不依赖底层 IO 或 JSON 异常类型。
- 项目不会把未经平台验证的“单 Cookie 最小集合”强行接入运行链路；认证所需 Cookie 由平台协议决定，避免错误裁剪导致随机掉登录。
- 日志与遥测不得记录 Cookie 原文；Web 会话令牌由独立的 session/CSRF 机制管理，不复用平台 Cookie。

**Q4: PII 检测的正则怎么设计？**

参考要点：
- 手机号：`(?<!\d)(?:\+?86[-\s]?)?1[3-9]\d{9}(?!\d)`，防前后多余数字
- 身份证：18 位，含年份/月份/日期校验
- 银行卡：62/45/52/37/35 开头，14-18 位
- 用 `(?<!\d)` 和 `(?!\d)` 防部分匹配

**Q5: defusedxml 解决什么问题？**

参考要点：
- 标准 `xml.etree.ElementTree` 有 XXE（XML External Entity）漏洞
- `defusedxml` 禁用外部实体解析
- 更新器解析 RSS/Atom feed 时用 `DefusedET`

**Q6: DomainPolicyEngine 如何防护 SSRF？覆盖了哪些危险地址段？DNS Rebinding 怎么防？**

参考要点（[shared/runtime_options.py#L114-L207](file:///d:/desktop/project/UniversalCrawlerProplus/shared/runtime_options.py#L114-L207)）：
- 引擎定位：单一 `DomainPolicyEngine` 类 + 模块级单例 `PUBLIC_DOMAIN_POLICY`（[L207](file:///d:/desktop/project/UniversalCrawlerProplus/shared/runtime_options.py#L207)），统一承接全项目的 SSRF 防护
- **`_is_unsafe_address`**（[L131-L142](file:///d:/desktop/project/UniversalCrawlerProplus/shared/runtime_options.py#L131-L142)）：静态方法，依据 `ipaddress` 模块判定任一 IP 是否"不可访问"，覆盖七类危险地址段：
  - `is_private`：RFC1918 私有地址（10/8、172.16/12、192.168/16）
  - `is_loopback`：127.0.0.0/8 与 ::1
  - `is_link_local`：169.254/16（云元数据服务典型攻击向量，如 AWS 169.254.169.254）
  - `is_multicast`、`is_reserved`、`is_unspecified`
  - `not is_global`：兜底拦截，覆盖 CGNAT 100.64.0.0/10 运营商级 NAT 段（测试 `test_domain_policy_rejects_credentials_and_non_global_cgnat` 验证）
- **`require_public_url`**（[L144-L187](file:///d:/desktop/project/UniversalCrawlerProplus/shared/runtime_options.py#L144-L187)）算法六步：
  1. 类型与空值校验（必须是字符串、非空）
  2. `urlsplit` 拆解，scheme 必须为 `http`/`https`（小写比较）
  3. **拒绝 URL 内嵌凭据**：`parts.username`/`parts.password` 不允许存在（防凭据泄露到日志/Referer）
  4. 主机名小写归一化，端口合法性校验
  5. 分支判定：主机是 IP 字面量 → 直接 `_is_unsafe_address(host)`；主机是域名 → `self._resolver or socket.getaddrinfo` 解析
  6. **DNS Rebinding 防护**：遍历 `getaddrinfo` 返回的**全部** addr_info，任一解析结果不安全即抛 `DomainPolicyViolation("禁止访问本地或内网地址")`，防止攻击者配置 DNS 返回混合地址（一个公网 + 一个内网）绕过
- resolver 可注入（`__init__` 参数 `resolver: Callable[..., list] | None`），便于单元测试 mock DNS，无需真实 DNS 依赖
- **`validate_redirect_response`**（[L189-L204](file:///d:/desktop/project/UniversalCrawlerProplus/shared/runtime_options.py#L189-L204)）：兼容 requests 库的 response-hook 协议（`hooks={"response": ...}`），在 `resolve_redirects` 跟随跳转前拦截；`*_args, **_kwargs` 兼容 requests 不同版本 hook 签名
- 策略启用条件：VideoItem 须 `meta["_network_policy"] == "public"`（[base.py#L753](file:///d:/desktop/project/UniversalCrawlerProplus/app/spiders/base.py#L753)、[downloaders/base.py#L64-L66](file:///d:/desktop/project/UniversalCrawlerProplus/app/core/downloaders/base.py#L64-L66)）才启用，既保证公网下载任务受保护，又允许内部任务（如本地工具下载）正常工作
- 调用拓扑：spider `_restricted_public_request_kwargs`（平台白名单+重定向双重校验）、`_safe_playwright_goto`（Playwright 导航前校验）、下载器 `_domain_policy_request_kwargs`（返回 `{"hooks":{"response": validate_redirect_response}}`）、CLI 直链 `validate_direct_download_url`

技术细节：七类危险地址判定（`not is_global` 兜底覆盖 CGNAT）+ DNS 全地址遍历防 Rebinding + 凭据 URL 拒绝 + resolver 可注入测试。单引擎多协议适配（requests hook / curl_cffi 手动循环 / Playwright 前置校验 / urllib 独立白名单）。

**Q7: `REDIRECT_STATUS_CODES` 为什么集中定义？HLS 代理为什么禁用库自动重定向改用手动循环？跨 origin 时怎么处理敏感头？**

参考要点：

  - `app/core/downloaders/m3u8.py:1781`：`_curl_cffi_session_response` 手动重定向循环
- **HLS 代理手动重定向循环**（[m3u8.py#L914-L961](file:///d:/desktop/project/UniversalCrawlerProplus/app/core/downloaders/m3u8.py#L914-L961)）：
  - 关键：`allow_redirects=False` 禁用 curl_cffi 自动跟随，自己实现重定向循环，确保**每跳都经过策略校验**

  ```python
  if self._url_origin(current_url) != self._url_origin(target_url):
      request_headers = {k:v for k,v in request_headers.items()
                         if k.lower() not in {"authorization","cookie","host","proxy-authorization"}}
  ```

  - `TrustedRedirectHandler(urllib.request.HTTPRedirectHandler)`：重写 `redirect_request`，在 urllib 跟随 30x 前调 `validate_asset_url`
  - `validate_asset_url`（[L1302-L1320](file:///d:/desktop/project/UniversalCrawlerProplus/app/services/secure_updater.py#L1302-L1320)）四重校验：强制 HTTPS、禁 localhost、禁裸 IP、主机必须在 `DEFAULT_ALLOWED_HOSTS`（github.com / objects.githubusercontent.com / release-assets.githubusercontent.com）

- 测试覆盖（[tests/unit/shared/test_runtime_helpers.py](file:///d:/desktop/project/UniversalCrawlerProplus/tests/unit/shared/test_runtime_helpers.py)）：DNS 失败、域名解析到 127.0.0.1、302 重定向到 127.0.0.1:8080、相对路径重定向放行、URL 内嵌凭据拒绝、CGNAT 拒绝

技术细节：集中定义 `frozenset` 避免散落硬编码；手动重定向循环（`allow_redirects=False`）是"每跳校验"的唯一可靠方式；跨 origin 剥离 4 类敏感头防凭据泄露；secure_updater 因协议栈不同独立实现 urllib 白名单（强制 HTTPS + 禁 localhost + 禁裸 IP + 主机白名单）。

**Q8: PII 递归 sanitize 的算法怎么实现？四种正则各自怎么设计？mask 回调的脱敏规则是什么？**

参考要点（[app/core/guardrails/pii_detection.py](file:///d:/desktop/project/UniversalCrawlerProplus/app/core/guardrails/pii_detection.py)）：
- **四种 PII 正则**（[L9-L16](file:///d:/desktop/project/UniversalCrawlerProplus/app/core/guardrails/pii_detection.py#L9-L16)），均用负向断言 `(?<!\d)`/`(?!\d)` 防部分匹配子串片段：
  - `PHONE_RE`（L9）：`(?<!\d)(?:\+?86[-\s]?)?1[3-9]\d{9}(?!\d)`——手机号，支持 +86 前缀和分隔符，1[3-9] 开头共 11 位
  - `ID_CARD_RE`（L10）：`(?<!\d)\d{6}(?:19|20)\d{2}(?:0[1-9]|1[0-2])(?:0[1-9]|[12]\d|3[01])\d{3}[\dXx](?!\d)`——身份证，6 位地址码 + 19/20 世纪年份 + 月日校验 + 3 位 + 校验位（支持 X/x）
  - `EMAIL_RE`（L11）：`(?<![\w.%-])[\w.%+-]+@[\w.-]+\.[A-Za-z]{2,}(?![\w.-])`——标准 RFC 风格
  - `BANK_CARD_RE`（L12-L16）：`(?<![\w/=&?#])(?:62[0-9]{14,17}|45[0-9]{12,15}|52[0-9]{14,17}|37[0-9]{11,13}|35[0-9]{14,17})(?![\w/=&?#])`——银行卡，按 BIN 段分：62(银联)/45(Visa)/52(Master)/37(Amex)/35(JCB)
- **递归 sanitize 算法**（[L21-L30](file:///d:/desktop/project/UniversalCrawlerProplus/app/core/guardrails/pii_detection.py#L21-L30)）：`sanitize(value: Any) -> Any` 按类型 dispatch：
  - `str` → 调 `sanitize_text` 做正则替换
  - `Mapping`（dict 等）→ 字典推导式 `{key: sanitize(item) for key, item in value.items()}`——**注意：key 不做 sanitize，只递归 value**
  - `list` → 列表推导式递归每个元素
  - `tuple` → 重建 tuple 递归每个元素
  - 其它（含 None/int/float/bool/自定义对象）→ 原样返回；**没有 `set`/`frozenset` 分支**，会原样返回（潜在盲点）
- **sanitize_text 实现**（[L33-L42](file:///d:/desktop/project/UniversalCrawlerProplus/app/core/guardrails/pii_detection.py#L33-L42)）：顺序对四种正则调 `re.Pattern.subn(pattern_callback, text)`，每次拿到 `(新文本, 替换次数)`，新文本传给下一轮正则
- **四个私有 mask 回调**（[L56-L83](file:///d:/desktop/project/UniversalCrawlerProplus/app/core/guardrails/pii_detection.py#L56-L83)）：
  - `_mask_phone`：取最后 11 位（处理 +86 前缀溢出），格式 `前3 + **** + 后4`（如 `138****8000`）
  - `_mask_id_card`：`前6 + 8 个 * + 后4`（如 `110105********333X`）
  - `_mask_email`：名段 ≤2 保留 1 位+`*`；>2 保留 2 位+`***`，域名全保留（如 `us***@example.com`）
  - `_mask_bank_card`：<14 位原样返回；否则 `前6 + ****** + 后4`（如 `622202******7890`）
- **公开 API**：仅 `sanitize`、`sanitize_text`、`get_masked_count`、`reset_masked_count` 四个；项目中**不存在** `sanitize_pii`/`mask_pii`/`detect_pii` 命名；`sanitize_text` 在生产代码中无直接调用点，仅用于测试
- **包导出**（[app/core/guardrails/__init__.py#L4](file:///d:/desktop/project/UniversalCrawlerProplus/app/core/guardrails/__init__.py#L4)）：`from app.core.guardrails.pii_detection import sanitize`，`__all__` 仅导出 `sanitize`（不含 `sanitize_text`/`get_masked_count`）；**不存在** `app/core/guardrails/pii.py`，PII 职责全部由 `pii_detection.py` 承担
- **易混淆点**：项目另有 `app/utils/filenames.py` 的 `sanitize_filename`（文件名清理，非 PII），在 download_manager/file_service/secure_updater 等多处调用——与 PII 脱敏无关

技术细节：递归类型 dispatch（str/Mapping/list/tuple 四类，key 不脱敏）+ 负向断言防子串误匹配 + BIN 段分类银行卡正则 + 顺序 subn 链式替换 + 四种 mask 规则（保留首尾、掩码中段）。

**Q6: 你项目里 `heapq.nlargest` 实际用在哪?为什么不用 `sorted+[:k]`?**

参考要点（`d:\desktop\project\UniversalCrawlerProplus\app\services\file_service.py#L284-L311`）:
- 实际用例：`MediaLibraryService.scan_directory(self, directory: str, max_scan_count: int = 1000)` 扫描下载目录
- 输入数据结构：`media_entries: list[tuple[float, str]]`（[file_service.py#L292](file:///d:/desktop/project/UniversalCrawlerProplus/app/services/file_service.py#L292)）—— `(st_mtime, filename)` 元组列表，通过 `os.scandir` + `entry.stat().st_mtime` 构建（[file_service.py#L293-L303](file:///d:/desktop/project/UniversalCrawlerProplus/app/services/file_service.py#L293-L303)）
- 截断分支（[file_service.py#L305-L311](file:///d:/desktop/project/UniversalCrawlerProplus/app/services/file_service.py#L305-L311)）：
  - `original_count = len(media_entries)`、`truncated = original_count > max_scan_count`
  - 截断分支：`heapq.nlargest(max_scan_count, media_entries, key=lambda item: item[0])`
  - 非截断分支仍走 `sorted(media_entries, key=lambda item: item[0], reverse=True)`
- 注释（[file_service.py#L308](file:///d:/desktop/project/UniversalCrawlerProplus/app/services/file_service.py#L308)）说明动机："大目录只保留最近更新的一部分文件，避免全量排序和前端渲染同时放大延迟"
- 截断结果通过 `ScanResult.truncated` / `original_count` 字段（[file_service.py#L26-L27](file:///d:/desktop/project/UniversalCrawlerProplus/app/services/file_service.py#L26-L27)）回传上层，由 `_build_scan_summary_message` 生成"⚠️ 文件过多 (N个)，仅加载最新的 M 个"提示
- 阈值来源：函数默认 `max_scan_count=1000`，运行时由 `MediaLibraryMixin._resolve_scan_limit` 从配置读取 `cfg.get("download", "local_scan_limit", 1000)`（[media_library_runtime.py#L63-L64](file:///d:/desktop/project/UniversalCrawlerProplus/app/services/media_library_runtime.py#L63-L64)）

技术细节：`heapq.nlargest(k, n)` 复杂度 `O(n log k)`，当 `k=1000` 远小于 `n`（如 50000 个文件的下载目录）时比 `sorted` 的 `O(n log n)` 显著更优；同时内部仅维护大小为 `k` 的最小堆，内存占用 `O(k)` 而非 `O(n)`。注意 `heapq.nlargest` 在 `k` 接近 `n` 时会退化为内排序（CPython 实现里 `k >= n` 直接 `sorted`），所以阈值 1000 在大目录场景下收益明显。截断后保留下的是 `st_mtime` 最大的 1000 个文件（即最近修改的），符合"最近下载优先展示"的产品语义。`heapq.nsmallest` 项目中未使用。
