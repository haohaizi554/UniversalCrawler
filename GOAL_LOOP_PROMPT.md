## 目标

把 GUI 设置中心和 WebUI 设置体验做成真实可用、热加载、用户友好且前后端一致的生产级配置中心。

## 强制要求

1. 设置前端必须使用选项控件；除了下载文件夹目录、自定义代理端口等明确需要用户输入的字段，其余设置项不允许自由键入。
2. 所有设置项必须真实接入配置、运行态和持久化，不允许只做视觉控件。
3. 修复不可用的 Switch 控件，保证点击、状态、样式和热加载都可用。
4. 字体大小、页面缩放、深浅色主题必须实时响应；右上角主题切换和设置面板主题选择必须双向同步。
5. 点击“绑定默认打开方式”必须有真实反应，GUI 和 WebUI 都要给出用户可感知结果。
6. 平台认证状态必须基于真实 cookie/session 检测；代理选项要显示主流软件名和端口，如 Clash、v2rayN、V2Ray、Qv2ray、sing-box、NekoRay、HTTP/SOCKS 自定义等。自定义代理允许输入端口，但要尽量降低用户理解成本。
7. 平台设置页面要消除截图中的大块空白，保持工业化、紧凑、对齐、美观。
8. 下载速率限制中 `0 KB/s` 必须明确显示为“无限制”。
9. 检查所有设置项是否真实实现，全局应用显示必须一致。
10. WebUI 所有页面的 UI、交互、状态和体验必须与 GUI 对齐，不能出现功能或视觉漂移。

## 执行循环

1. 先读取当前代码、截图和测试，建立现状清单。
2. 为每条要求找到权威验证方式：代码、运行态、测试、截图或 API 结果。
3. 修改代码时优先复用现有 `FrontendStateService`、`SettingsPage`、WebUI 静态页面和配置模型。
4. 每次修复后运行针对测试；关键路径完成后运行 GUI/WebUI smoke 和全量测试。
5. 若发现具有教学意义的问题，在 `docx/` 下补充 markdown 记录。
6. 完成前进行逐项审计；证据不足就继续修，不以“看起来差不多”作为完成。
